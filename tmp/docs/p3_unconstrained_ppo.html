一 · P3 是什么？它的作用 — What is P3 and Why?
中 P3 是一个"参考基线"，不是实际可用的部署方法。

它回答一个问题：如果 PPO 完全不理会任何约束（容量限制、冲突限制），能达到多高的 AR？

这个数字本身没有实际意义（因为违规了），但它告诉我们：
  • 约束对 AR 的"压低程度"有多大（P3 AR 减去 P4 AR 就是约束的代价）
  • 如果约束处理做得好（如 P4 掩码），AR 最高能接近多少
  • 验证 RL 框架本身是否正常工作（无约束情况下 AR 应该很高）
EN P3 is a reference baseline, not a usable deployment method.

It answers: If PPO completely ignores all constraints (capacity and conflict), how high can AR get?

The number itself is not meaningful (violations make it invalid), but it tells us:
  • How much constraints "suppress" AR (P3 AR minus P4 AR = cost of constraints)
  • How close to an unconstrained ceiling a well-constrained method (P4) can get
  • Whether the RL framework itself is working (unconstrained AR should be high)

EN P3's step logic (key parts): 
def step(action):
    svc = services[current_step]
    cap_violated      = remaining_vms[action] < svc.requirement
    conflict_violated = has_conflict(action, current_step)

    # Violations only counted, no penalty whatsoever
    if cap_violated:
        capacity_violations += 1
    if conflict_violated:
        conflict_violations += 1

    # Regardless of violation, compute and accumulate utilization
    ru = svc.requirement / initial_vms[action]
    remaining_vms[action] -= svc.requirement   # can go negative!
    total_ru += ru                              # violation steps also add to AR

    # Update AR (active ECUs = those with at least one service)
    active = count of ECUs with at least one service
    ar = total_ru / active

    # Terminal bonus: always +AR regardless of violations
    if done:
        terminal_bonus = ar

def action_masks():
    svc = services[当前步]
    mask = (remaining_vms >= svc.requirement)  # 只看容量，不看冲突！
    if mask 全为 False:
        best = argmax(remaining_vms)
        mask[best] = True   # 兜底：选容量最大的
    return mask

四 · 奖励函数：AR 虚高的根本原因 — Reward Function
中 P3 的每步奖励：
\[ r_t = \frac{\mathrm{req}_i}{\mathrm{cap}_j} \quad \text{（不管有没有违规，都这么算）} \]
终局奖励：
\[ r_{\text{terminal}} = AR_{\text{最终}} \quad \text{（始终 +AR，无论违规多少次）} \]
奖励信号对 AI 的引导：
由于违规没有任何代价，AI 学会的策略是"哪台 ECU 利用率最高就放哪台"，完全不在乎约束。 这使得 AI 在同一台 ECU 上堆叠大量服务，利用率飙升（远超1.0），AR 虚高。
EN P3 per-step reward:
\[ r_t = \frac{\mathrm{req}_i}{\mathrm{cap}_j} \quad \text{(computed regardless of violations)} \]
Terminal reward:
\[ r_{\text{terminal}} = AR_{\text{final}} \quad \text{(always +AR regardless of violations)} \]
What the AI learns:
Since violations have zero cost, the AI learns "maximize utilization ratio regardless of constraints." It stacks many services onto a few ECUs, pushing utilization far above 1.0, inflating AR artificially.

六 · 为什么 LT 场景 AR 会超过 1？— Why AR Exceeds 1 in LT
中 正常情况下 AR 应该 ≤ 1（利用率 = 已用/总容量，最大值是 1）。
但 P3 的 LT 场景实验 AR 高达 11.8——这是怎么发生的？

根本原因：容量可以变成负数。
例如：ECU j 容量 = 100，但 P3 允许把 200 的需求放进去（违规）。
此时 remaining_vms[j] = 100 - 200 = -100（负数）。
但利用率计算用的是初始容量：
\[ \text{ECU j 的利用率} = \frac{\sum_{\text{放在j的服务}} \mathrm{req}_i}{\mathrm{cap}_j} = \frac{200}{100} = 2.0 \]
利用率超过 1.0！多个 ECU 都这样叠加后，AR 轻易超过 1，甚至达到 11.8。
这个 AR 数值完全没有实际意义——它对应的部署方案违反了容量约束，在现实中根本无法运行。
EN Normally AR should be ≤ 1 (utilization = used / capacity, max = 1).
But P3's LT scenario AR reaches 11.8 — how?

Root cause: remaining capacity can go negative.
Example: ECU j has capacity = 100, but P3 allows 200 units of demand to be placed (violation).
Then remaining_vms[j] = 100 - 200 = -100 (negative).
But utilization uses the initial capacity:
\[ \text{ECU j utilization} = \frac{\sum_{\text{services on j}} \mathrm{req}_i}{\mathrm{cap}_j} = \frac{200}{100} = 2.0 \]
Utilization exceeds 1.0! With multiple ECUs doing this, AR easily exceeds 1, reaching 11.8.
This AR value is entirely meaningless — the corresponding deployment violates capacity constraints and cannot run in reality.
