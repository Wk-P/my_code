一 · 什么是整数线性规划？— What is ILP?
中 先说"线性规划（LP）"：
假设你有 100 元，可以买苹果（1元/个）和香蕉（2元/个），想让总水果数最多。这是一个"目标"（最多水果）+ "约束"（不超 100 元）的优化问题，而且目标和约束都是"加减乘除"的简单线性关系——这就叫线性规划。

整数线性规划（ILP） 额外要求：变量只能取 整数（比如0或1），不能是 0.5 个苹果。 加了整数要求，问题难度会从"随便求解"跳升到"理论上很难（NP-hard）"。
EN First, "Linear Programming (LP)":
Imagine you have $100 to spend on apples ($1 each) and bananas ($2 each), wanting to maximize the total number of fruits. This is an optimization with an objective (most fruits) and a constraint (≤$100), where both are simple linear relationships — that's LP.

Integer Linear Programming (ILP) adds one rule: variables must be integers (e.g., 0 or 1), not fractions like 0.5 apples. This seemingly small restriction jumps the difficulty from "easy" to "NP-hard" in theory.
中 在 ECU 部署中，ILP 的角色：
我们有 N 台 ECU（电脑/控制单元）和 M 个服务（程序），要决定"每个服务装到哪台 ECU 上"。 每个决策只有两种结果：放（1）或不放（0）。 ILP 会在所有可能的放置方案里，找到平均资源利用率（AR）最高的那个——这是数学意义上的全局最优解，所以 ILP 的结果是本项目所有算法的性能上界。

缺点：当 M 和 N 较大时，计算时间指数级增长，只适合作为"标准答案"来参照，不能实时使用。
EN ILP's role in ECU deployment:
We have N ECUs and M services, deciding "which ECU gets each service." Each decision is binary: assign (1) or not (0). ILP searches all possible assignments to find the one with the highest average resource utilization (AR) — this is the global optimum, making ILP the performance upper bound for all algorithms in this project.

Drawback: computation time grows exponentially with M and N; it's only feasible as a reference "ground truth," not for real-time use.
二 · 决策变量：把"放还是不放"变成数字 — Decision Variables
中 变量 \(x_{ij}\)：服务 i 放到 ECU j 吗？
\[ x_{ij} \in \{0,\, 1\}, \quad i=1\ldots M,\; j=1\ldots N \]
  • \(x_{ij} = 1\)：服务 i 放到 ECU j 上
  • \(x_{ij} = 0\)：服务 i 没有放在 ECU j 上
共有 M × N 个这样的变量。例如 M=15，N=10，就有 150 个 0/1 变量。
EN Variable \(x_{ij}\): does service i go on ECU j?
\[ x_{ij} \in \{0,\, 1\}, \quad i=1\ldots M,\; j=1\ldots N \]
  • \(x_{ij} = 1\): service i is assigned to ECU j
  • \(x_{ij} = 0\): service i is NOT assigned to ECU j
There are M × N such variables. For M=15, N=10, that's 150 binary variables.
中 变量 \(y_j\)：ECU j 有服务在上面吗（"活跃"了吗）？
\[ y_j \in \{0,\, 1\}, \quad j=1\ldots N \]
  • \(y_j = 1\)：ECU j 上至少有一个服务 → ECU j "活跃"
  • \(y_j = 0\)：ECU j 上没有任何服务 → ECU j "空闲"
为什么需要 \(y_j\)？因为计算 AR 时，分母是"活跃 ECU 的数量"，需要知道哪些 ECU 是活跃的。
EN Variable \(y_j\): is ECU j "active" (has at least one service)?
\[ y_j \in \{0,\, 1\}, \quad j=1\ldots N \]
  • \(y_j = 1\): ECU j hosts at least one service → "active"
  • \(y_j = 0\): ECU j hosts nothing → "idle"
Why do we need \(y_j\)? Because the AR formula divides by the number of active ECUs, so we must track which ECUs are active.
三 · 四个约束条件 — Four Constraints
中 约束 1：每个服务恰好放一次（不能漏放，也不能分身）
\[ \sum_{j=1}^{N} x_{ij} = 1, \quad \forall i \]
读法：对于每个服务 i，把它放到各台 ECU 的决策加起来恰好等于 1。 也就是说，服务 i 只能在恰好一台 ECU 上取 1，其余 ECU 上都取 0。
EN Constraint 1: each service assigned exactly once (no omission, no splitting)
\[ \sum_{j=1}^{N} x_{ij} = 1, \quad \forall i \]
Reading it: for each service i, the sum of all placement decisions across all ECUs must equal 1. Service i is placed on exactly one ECU; the rest are 0.
中 约束 2：每台 ECU 的总容量不超限
\[ \sum_{i=1}^{M} x_{ij} \cdot \mathrm{req}_i \;\leq\; \mathrm{cap}_j, \quad \forall j \]
\(\mathrm{req}_i\) 是服务 i 需要的资源量，\(\mathrm{cap}_j\) 是 ECU j 的总容量。
直觉：把放在 ECU j 上的所有服务需求加起来，不能超过 ECU j 的"装载上限"。就像行李箱装不下太多东西。
EN Constraint 2: no ECU exceeds its capacity
\[ \sum_{i=1}^{M} x_{ij} \cdot \mathrm{req}_i \;\leq\; \mathrm{cap}_j, \quad \forall j \]
\(\mathrm{req}_i\) = resources needed by service i; \(\mathrm{cap}_j\) = total capacity of ECU j.
Intuition: the sum of resource demands for all services placed on ECU j must not exceed j's capacity — like a suitcase that can only hold so much.
中 约束 3：冲突服务不能同台（同一冲突集最多1个）
\[ \sum_{i \in C} x_{ij} \;\leq\; 1, \quad \forall C \in \mathcal{C},\; \forall j \]
\(\mathcal{C}\) 是所有冲突集的集合，\(C\) 是其中一个冲突集（一组互相打架的服务）。
直觉：有些服务互相冲突（比如争同一个端口），同一冲突集里的服务，在同一台 ECU 上最多只能有一个。
EN Constraint 3: conflicting services cannot share the same ECU
\[ \sum_{i \in C} x_{ij} \;\leq\; 1, \quad \forall C \in \mathcal{C},\; \forall j \]
\(\mathcal{C}\) is the set of all conflict groups; \(C\) is one conflict group (services that clash with each other).
Intuition: some services conflict (e.g., competing for the same port). At most one service from each conflict group can be hosted on any single ECU.
中 约束 4：活跃变量 \(y_j\) 与 \(x_{ij}\) 保持一致
\[ x_{ij} \leq y_j \quad \forall i,j \qquad \text{（有服务才算活跃）} \] \[ \sum_{i=1}^{M} x_{ij} \geq y_j \quad \forall j \qquad \text{（活跃必须有服务）} \]
第一行：如果某服务放在 ECU j（\(x_{ij}=1\)），那 ECU j 必须标为活跃（\(y_j=1\)）。
第二行：如果 ECU j 被标为活跃（\(y_j=1\)），上面至少有一个服务。
这两行合在一起，确保 \(y_j=1 \Leftrightarrow\) ECU j 上有至少一个服务。
EN Constraint 4: \(y_j\) correctly tracks whether ECU j is active
\[ x_{ij} \leq y_j \quad \forall i,j \qquad \text{(service implies active)} \] \[ \sum_{i=1}^{M} x_{ij} \geq y_j \quad \forall j \qquad \text{(active implies at least one service)} \]
Line 1: if any service is placed on ECU j (\(x_{ij}=1\)), ECU j must be marked active (\(y_j=1\)).
Line 2: if ECU j is marked active, it must host at least one service.
Together: \(y_j=1 \Leftrightarrow\) ECU j hosts ≥1 service.
四 · 目标函数 AR：为什么是"分式"难题 — The Fractional Objective
中 AR 的定义：
\[ AR = \frac{1}{|\{j : y_j=1\}|} \sum_{\{j : y_j=1\}} \frac{\displaystyle\sum_{i : x_{ij}=1} \mathrm{req}_i}{\mathrm{cap}_j} \]
用大白话说：
  1. 对每台有服务的 ECU，计算它的利用率 = 已用容量 ÷ 总容量
  2. 把所有活跃 ECU 的利用率加起来
  3. 除以活跃 ECU 的数量，得到平均值
AR 越高，说明 ECU 的资源浪费越少，部署越"紧凑"。
EN AR definition:
\[ AR = \frac{1}{|\{j : y_j=1\}|} \sum_{\{j : y_j=1\}} \frac{\displaystyle\sum_{i : x_{ij}=1} \mathrm{req}_i}{\mathrm{cap}_j} \]
In plain language:
  1. For each ECU that has at least one service, compute its utilization = used capacity ÷ total capacity
  2. Sum those utilization values across all active ECUs
  3. Divide by the number of active ECUs to get the average
Higher AR = less wasted capacity = more compact deployment.
中 为什么这是"难题"？
把 AR 写成决策变量的函数：
\[ AR = \frac{\underbrace{\displaystyle\sum_{i,j} x_{ij} \cdot \frac{\mathrm{req}_i}{\mathrm{cap}_j}}_{F(\mathbf{x})\;\text{—利用率之和}}} {\underbrace{\displaystyle\sum_j y_j}_{G(\mathbf{y})\;\text{—活跃ECU数}}} \]
分子 \(F(\mathbf{x})\) 和分母 \(G(\mathbf{y})\) 都是决策变量的函数。
这种"分子分母都变化"的目标叫分式规划（Fractional Programming），不能直接用标准 LP 求解器处理。
类比：想最大化"每台被占用机器的平均效率"——你同时要决定开几台机器（分母）和怎么分配任务（分子），这两件事互相纠缠。
EN Why is this hard?
Writing AR as a function of decision variables:
\[ AR = \frac{\underbrace{\displaystyle\sum_{i,j} x_{ij} \cdot \frac{\mathrm{req}_i}{\mathrm{cap}_j}}_{F(\mathbf{x})\;\text{— sum of utilizations}}} {\underbrace{\displaystyle\sum_j y_j}_{G(\mathbf{y})\;\text{— number of active ECUs}}} \]
Both numerator \(F(\mathbf{x})\) and denominator \(G(\mathbf{y})\) are functions of the decision variables.
This "both-sides-change" objective is called Fractional Programming and cannot be solved directly by standard LP solvers.
Analogy: maximizing "average efficiency per active machine" while simultaneously deciding how many machines to use (denominator) and how to distribute tasks (numerator) — the two decisions are entangled.
五 · Dinkelbach 算法：化分式为线性 — Dinkelbach's Algorithm
中 核心数学思想：
Dinkelbach（1967年）证明了一个定理：
最大化 \(F/G\) 等价于 找到参数 \(\lambda^*\) 使得：
\[ \max_{\mathbf{x},\mathbf{y}} \bigl[ F(\mathbf{x}) - \lambda^* \cdot G(\mathbf{y}) \bigr] = 0 \]
关键洞察：\(F/G > \lambda\) 当且仅当 \(F - \lambda G > 0\)（两边乘以正的 \(G\)）。
所以"找最大 AR"变成了"找合适的 \(\lambda\)"——而对于固定的 \(\lambda\)，\(F - \lambda G\) 是线性函数，标准 ILP 就能处理！
EN Core mathematical idea:
Dinkelbach (1967) proved: maximizing \(F/G\) is equivalent to finding parameter \(\lambda^*\) such that:
\[ \max_{\mathbf{x},\mathbf{y}} \bigl[ F(\mathbf{x}) - \lambda^* \cdot G(\mathbf{y}) \bigr] = 0 \]
Key insight: \(F/G > \lambda\) if and only if \(F - \lambda G > 0\) (multiply both sides by positive \(G\)).
So "maximize AR" becomes "find the right \(\lambda\)" — and for a fixed \(\lambda\), \(F - \lambda G\) is linear, solvable by standard ILP!
中 \(\lambda\) 的直觉含义：
\(\lambda\) 可以理解为"每新增一台活跃 ECU 需要付出的'利用率代价'"。
  • 如果一台新 ECU 带来的利用率增量 > \(\lambda\)，就值得开它
  • 如果带来的增量 < \(\lambda\)，不如少开一台 ECU，把服务塞进已有的
  • 当 \(\lambda = AR^*\) 时，增量恰好等于当前平均水平，系统达到最优均衡
EN Intuition for \(\lambda\):
\(\lambda\) is the "utilization threshold for activating one more ECU."
  • If a new ECU brings utilization gain > \(\lambda\), it's worth activating
  • If the gain < \(\lambda\), better to pack services onto fewer ECUs
  • When \(\lambda = AR^*\), the gain exactly equals the current average — optimal equilibrium
中 迭代步骤（每步说明）：
1
初始化 \(\lambda = 0\)（假设每台 ECU 的"代价"为零，即先不考虑开多少台）
2
解线性化子问题：
\[ \max \sum_{i,j} x_{ij}\frac{\mathrm{req}_i}{\mathrm{cap}_j} - \lambda \sum_j y_j \]
这是一个标准 0/1 整数规划，CBC 求解器可以处理。得到当前最优放置方案 \(\mathbf{x}^*, \mathbf{y}^*\)。
3
计算新的 \(\lambda\)：
\[ \lambda_{\text{new}} = \frac{F(\mathbf{x}^*)}{G(\mathbf{y}^*)} = \frac{\text{总利用率}}{\text{活跃ECU数}} = AR_{\text{当前方案}} \]
4
检查收敛：若 \(|\lambda_{\text{new}} - \lambda| < 10^{-6}\)，停止；否则令 \(\lambda \leftarrow \lambda_{\text{new}}\) 回到第2步。最多迭代20次。
EN Iteration steps (explained one by one):
1
Initialize \(\lambda = 0\) (assume activating ECUs has no cost — ignore the denominator for now)
2
Solve the linearized sub-problem:
\[ \max \sum_{i,j} x_{ij}\frac{\mathrm{req}_i}{\mathrm{cap}_j} - \lambda \sum_j y_j \]
This is a standard 0/1 ILP that CBC can handle. Get the current optimal assignment \(\mathbf{x}^*, \mathbf{y}^*\).
3
Update \(\lambda\):
\[ \lambda_{\text{new}} = \frac{F(\mathbf{x}^*)}{G(\mathbf{y}^*)} = \frac{\text{total utilization}}{\text{active ECU count}} = AR_{\text{current plan}} \]
4
Check convergence: if \(|\lambda_{\text{new}} - \lambda| < 10^{-6}\), stop; otherwise set \(\lambda \leftarrow \lambda_{\text{new}}\) and go back to step 2. Maximum 20 iterations.
中 为什么收敛且正确？
每次迭代后，\(\lambda\) 值单调递增并趋向真实 AR 最优值。证明思路：
  • 新方案在目标函数 \(F - \lambda G\) 下是最优的
  • 如果旧 \(\lambda\) 比真实 AR* 小，新方案会把更多利用率"挤进来"，使 \(\lambda_{\text{new}}\) 更接近真实值
  • 当 \(\lambda^* = AR^*\) 时，子问题最优值为 0，算法收敛
实践中通常 3–8 次迭代就收敛，精度优于 \(10^{-6}\)。
EN Why does it converge correctly?
After each iteration, \(\lambda\) increases monotonically toward the true optimal AR. The argument:
  • The new plan is optimal under objective \(F - \lambda G\)
  • If the old \(\lambda\) is below true AR*, the new plan squeezes in more utilization, pushing \(\lambda_{\text{new}}\) closer to the true value
  • When \(\lambda^* = AR^*\), the sub-problem optimal value is 0 — convergence
In practice, 3–8 iterations typically suffice to converge to precision < \(10^{-6}\).
六 · CBC 求解器：搜索整数最优解 — The CBC Solver
中 CBC（Coin-or Branch and Cut）是 Dinkelbach 每轮迭代中用来求解 0/1 ILP 的求解器。它用的核心方法是分支定界（Branch and Bound）。

分支定界的三步：
1
松弛（Relax）：先放宽整数要求，把 \(x_{ij} \in \{0,1\}\) 改成 \(0 \leq x_{ij} \leq 1\)（允许小数）。这时问题是简单的 LP，可以快速求解，得到的最优值是真实最优的上界（松弛后值更大或相等）。
2
分支（Branch）：如果松弛解有小数（如 \(x_{12} = 0.7\)），就分成两个子问题：一个强制 \(x_{12} = 0\)，另一个强制 \(x_{12} = 1\)，分别继续求解。这样形成一棵搜索"树"。
3
剪枝（Prune）：如果某个子问题的上界比已知最好整数解还差，就不再展开它（剪掉这个分支）。这样大大减少搜索量。
EN CBC (Coin-or Branch and Cut) is the solver that handles the 0/1 ILP in each Dinkelbach iteration. Its core method is Branch and Bound.

Three steps of branch and bound:
1
Relax: remove the integrality requirement, allowing \(0 \leq x_{ij} \leq 1\) (fractions OK). The resulting LP is easy to solve and gives an upper bound on the true ILP optimum.
2
Branch: if the relaxed solution has a fractional variable (e.g., \(x_{12} = 0.7\)), create two sub-problems: one forcing \(x_{12} = 0\), another forcing \(x_{12} = 1\). Recurse. This builds a search tree.
3
Prune: if a sub-problem's upper bound is worse than the best integer solution found so far, discard that branch. This dramatically reduces the search space.
中 为什么慢？
最坏情况下，搜索树的节点数是指数级的（2M×N个可能的 0/1 组合）。对于 M=15, N=10，有 2150 种可能——天文数字。虽然剪枝大大减少了实际搜索量，但在 200 个测试场景下依然很慢。

为什么还要用？
因为 ILP 是目前已知能找到数学上全局最优解的方法，它的 AR 结果是所有 RL 算法的性能天花板。
EN Why is it slow?
In the worst case, the search tree has an exponential number of nodes (2M×N combinations). For M=15, N=10 that's 2150 — an astronomical number. Pruning helps greatly, but it's still slow over 200 test scenarios.

Why use it at all?
Because ILP is the only method that provably finds the global mathematical optimum. Its AR result serves as the performance ceiling for all RL algorithms.
