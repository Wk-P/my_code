P7 Sequential PPO (SeqPPO)

联合选择"哪个服务 + 哪个 ECU"的顺序决策方法 · Joint service-selection & ECU-assignment via sequential RL

一、核心思想 — 为什么要"自由选择顺序"？

P4（MaskablePPO）每一步只决定"把当前服务放到哪个 ECU"—— 服务的处理顺序是固定排好的（通常按资源需求从大到小排序）。 这意味着 AI 只能在给定顺序下局部最优，但顺序本身不可更改。

P7 打破这个限制：每一步 AI 同时决定 先处理哪个服务（从所有未放置服务中选） AND 放到哪个 ECU。 动作空间从 N 扩展到 M × N。

直觉类比：P4 = 按固定菜单点菜，每次只选餐厅； P7 = 自由搭配，既选今天吃什么，也选去哪家餐厅。

P4 MaskablePPO

动作空间 = Discrete(N)

每步：选 ECU（服务顺序固定）

状态："下一个要放的服务已确定"

P7 SeqPPO

动作空间 = Discrete(M × N)

每步：同时选服务 + 选 ECU

状态："所有未放置服务均可选"

二、动作空间设计 — M × N 编码

如果有 M=15 个服务、N=10 个 ECU，动作空间共有 15 × 10 = 150 个动作，每个动作用一个整数 a 表示。

svc_idx = a // N　　　（整除，得到服务编号，0 ~ M-1）
ecu_idx = a % N　　　（取余，得到 ECU 编号，0 ~ N-1）

1010服务1 → ECU0
2323服务2 → ECU3
149149服务14 → ECU9

三、动作掩码 — 双重合法性检查

P7 的动作掩码要同时满足两个条件，才允许执行动作 a：

  1. 容量约束 目标 ECU 剩余容量 ≥ 服务需求资源量
  2. 冲突约束 目标 ECU 未被列为该服务的禁止 ECU

特别注意：若服务 svc_idx 已放置（placed[svc_idx] == True）， 则该服务对应的所有 M × N 动作全部被屏蔽，不允许重复放置。 

# 伪代码：action_masks() 的核心逻辑
def action_masks(self):
    mask = [False] * (self.M * self.N)
    for svc_idx in range(self.M):
        if self.placed[svc_idx]:   # 已放置，整行屏蔽
            continue
        for ecu_idx in range(self.N):
            cap_ok  = self.remaining_vms[ecu_idx] >= self.reqs[svc_idx]
            conf_ok = ecu_idx not in self.forbidden[svc_idx]
            if cap_ok and conf_ok:
                mask[svc_idx * self.N + ecu_idx] = True
    return mask

四、观测空间 — 4 + 4N + 3M 维向量

AI 每步能"看到"的信息被编码为一个数值向量。 P7 的观测比 P4 多了每个服务的状态信息（3M 维），因此维度更高。

全局 (4 维)
① 当前 AR（平均资源利用率）
② 所有 ECU 剩余总容量（归一化）
③ 所有未放置服务的总需求（归一化）
④ 剩余未放置服务比例
每个 ECU (4N 维)
对每个 ECU j（j=0..N-1）：
① 初始容量/最大初始容量
② 剩余容量/初始容量
③ 允许放置的服务比例（1 - 冲突服务数/M）
④ 已放置服务数/M
每个服务 (3M 维)
对每个服务 i（i=0..M-1）：
① 资源需求/最大需求（归一化）
② 是否已放置（0 或 1）
③ 可合法放置的 ECU 比例

以 M=15, N=10 为例：总观测维度 = 4 + 4×10 + 3×15 = 4 + 40 + 45 = 89 维。 相比 P4（约 44 维），P7 的状态更丰富，因为 AI 需要主动选择下一个服务。

五、奖励函数 — 违规重罚 + 完美完成大奖励

每步奖励（Step Reward）

如果动作合法（无容量/冲突违规）：
    r_step = req_i / cap_j　　（本次放置的利用率贡献）

如果动作非法（产生违规）：
    r_step = -2.0　　（重罚）

终止奖励（Terminal Reward）

如果整个 episode 零违规：
    r_terminal = +AR　　（鼓励高利用率完美方案）

如果存在任何违规：
    r_terminal = +0.1 × AR　　（轻微奖励，主要信号已被 -2.0 抵消）

为什么终止奖励如此重要？

仅靠每步 req/cap 奖励，AI 可能学会"放置服务就能得分，违规无所谓"。 终止奖励把整个 episode 的结果与全局 AR 绑定—— 零违规才能拿到完整奖励， 引导 AI 从全局角度规划每一步动作，而非贪婪地最大化当前步奖励。

六、训练流程（与 P4 共享超参数）

  • 1
    初始化环境（40 个并行环境）
    每个环境独立采样一个场景（M 个服务的需求量 + 约束矩阵）， 所有服务标记为未放置（placed = [False] * M）。
  • 2
    采集经验（256 步/环境）
    每步 AI 调用 action_masks() 获取合法动作集合， Policy 网络在合法动作上 softmax 采样，执行，记录 (observation, action, reward, done, mask) 元组。
  • 3
    计算 GAE 优势估计
    用 Critic 网络估值，反向传播 GAE（λ=0.95, γ=0.999）， 得到每步的优势函数 Â_t。
  • 4
    PPO Clip 更新（10 个 epoch）
    计算概率比 r_t = π_new/π_old， Clip 在 [1-ε, 1+ε]（ε=0.2）内限制更新幅度， 同时更新 Actor（策略）和 Critic（价值）。
  • 5
    重复直到 500 万总步数
    不断迭代采集 → 更新 → 清空 buffer， 共约 500万 / (40 × 256) ≈ 488 次大迭代。

七、placed[] 追踪机制 — 防止重复放置

P7 用一个布尔数组 placed[0..M-1] 追踪哪些服务已放置。 每次 AI 成功放置服务 svc_idx 后：

执行动作 a
→
解码 svc_idx, ecu_idx
→
更新 remaining_vms[ecu_idx]
→
placed[svc_idx] = True
→
更新 action_masks()

当 sum(placed) == M 时，所有服务均已放置，episode 结束， 触发终止奖励。

八、P7 vs P4 — 自由选顺序为何反而更难？

直觉上，"自由选择顺序"应该能找到更好的方案。 但实验结果显示 P7 在 LT 场景下表现反而不如 P4—— 冲突违规率高达 85%，而 P4 为 0%。

原因分析：探索空间爆炸

  • 动作空间从 N=10 扩展到 M×N=150， 每步需要从 150 个动作中选择，合法动作的比例在 LT 场景下极低， 梯度信号稀疏，学习效率大幅下降。
  • 冲突约束的复杂交互： LT 场景（15 服务 / 10 ECU）资源极度紧张， 服务与 ECU 之间的冲突约束使得可行解空间极小； P7 需要同时学习"先放哪个"和"放哪里"， 这两个决策的耦合使得 5M 步训练远远不够收敛。
  • P4 受益于问题结构： 固定的服务排序（按需求大小）实际上是一个很好的启发式先验， 把一个难问题转化为一系列相对简单的 N 选 1 子问题。

P4 的优势

固定顺序 = 内置启发式先验

每步只需从 N=10 个动作选

LT: 零违规，AR=0.673

P7 的挑战

无先验，需自学最优顺序

每步从 M×N=150 个动作选

LT: 冲突违规率 85%，AR=0.571

十、关键洞察 — P7 的价值与局限

P7 的优势

  • 在 EQ 和 GT 场景（资源充裕）下实现零违规，证明可行性
  • 框架更通用：不依赖固定的服务排序启发式
  • 理论上拥有更大的解空间，可能找到 P4 无法发现的方案

P7 的局限（LT 场景）

  • 动作空间扩大 15 倍（从 10 → 150），需要更多训练步数才能收敛
  • 冲突违规率 85%，说明 5M 步训练远不足以在 LT 约束下掌握顺序选择
  • AR 仅 0.571，低于 P4（0.673）和 DDQN（0.659）

如果要改进 P7？

  • 增加训练步数（例如 20M～50M 步），给 AI 足够时间探索更大的动作空间
  • 引入课程学习（Curriculum Learning）：先在小问题（M=5, N=5）上训练，再逐步增大规模
  • 结合 P5 的 Lagrangian 惩罚：给违规动作额外惩罚，加速学习可行策略
  • 分层决策：先选服务（M 路 softmax），再选 ECU（N 路 softmax），减少耦合

##################################################
## P7 SeqPPO — 联合选服务+ECU 的 MaskablePPO
##################################################

# 环境初始化
def reset(env):
    env.placed = [False] * M          # 所有服务未放置
    env.remaining_vms = initial_cap.copy()
    return build_obs(env)              # 返回 4 + 4N + 3M 维观测

# 每步交互
def step(env, action):
    svc_idx = action // N
    ecu_idx = action % N

    if is_violation(svc_idx, ecu_idx):   # 容量或冲突违规
        reward = -2.0
        env.viol_count += 1
    else:
        env.remaining_vms[ecu_idx] -= env.reqs[svc_idx]
        reward = env.reqs[svc_idx] / env.caps[ecu_idx]  # req/cap

    env.placed[svc_idx] = True
    done = sum(env.placed) == M

    if done:
        AR = calc_AR(env)
        if env.viol_count == 0:
            reward += AR          # 完美完成：+AR
        else:
            reward += 0.1 * AR   # 有违规：仅 +0.1*AR

    return build_obs(env), reward, done

# PPO 训练主循环（与 P4 完全相同）
for step in range(5_000_000):
    masks = env.action_masks()           # M×N 维布尔掩码
    action = policy.sample(obs, masks)   # 仅在合法动作上采样
    obs, reward, done = env.step(action)
    buffer.store(obs, action, reward, done, masks)

    if buffer.full():
        advantages = compute_GAE(buffer, gamma=0.999, lam=0.95)
        for _ in range(10):              # 10 epoch PPO 更新
            ppo_clip_update(buffer, advantages, epsilon=0.2)
        buffer.clear()
