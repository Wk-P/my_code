1 Docstring — The Environment is Identical

"""
DDQN Environment — heavy penalty for violations + terminal bonus.

N < M: each ECU hosts multiple services.

Identical constraint-enforcement strategy to DQNEnv.  The algorithm
distinction (dual Q-networks to reduce overestimation) is in the trainer,
not the environment.

Constraints:
    - Capacity violation → heavy penalty (-2.0), placement still proceeds.
    - Conflict violation → heavy penalty (-2.0), placement still proceeds.
    - A well-trained agent should achieve zero violations on feasible scenarios.

Terminal bonus (lt-specific adaptation, aligns reward scale with PPO methods):
    +AR if zero violations, -AR if any violations.
    Episode always runs M steps (no early termination).
"""
Key Point

As stated in the docstring: "Identical constraint-enforcement strategy to DQNEnv". DDQNEnv's environment logic is identical to DQNEnv. The DDQN algorithmic improvement (dual network to reduce overestimation) happens in the Trainer, not the environment.

The docstring explicitly states the environment is identical to DQNEnv. The algorithm difference (Double Q-network update rule) lives entirely in the trainer, not the env.

3 Why Same Env, Different Results?

Answer: The Q-value update rule is different!

NoneDQN

DQN Target Calculation
target = r + γ × maxa Qtarget(s', a)

# target network handles both:
# 1. select optimal action (argmax)
# 2. evaluate the action's value (Q value)
# one network does both → overestimation bias!

NoneDDQN

DDQN Target Calculation
a* = argmaxa Qonline(s', a) ← online network selects action
target = r + γ × Qtarget(s', a*) ← target network evaluates value

# separate "selection" and "evaluation" responsibilities
# two independent networks check each other → reduces overestimation!

DQN —

Target network Q_target(s', a)
↓
max_a → select action with highest value
↓
same network evaluates action Q value
↓
prone to selecting overestimated actions due to noise

DDQN —

Online network Q_online(s', a)
↓
argmax → select best action a*
↓
target network evaluates Q_target(s', a*)
↓
target network "corrects" the online network's optimistic estimate

4 Concrete Overestimation Example

Scenario: next state s' has two possible actions a1, a2. Due to network estimation noise:

1
True Q values (ideal): Q(s', a1) = 0.85，Q(s', a2) = 0.80
2
Noisy network estimates: Q_hat(s', a1) = 0.90, Q_hat(s', a2) = 0.78 (a1 overestimated!)
DQN

same network evaluates → target = 0.90 (overestimated! true is 0.85)
Result: network learns falsely optimistic Q values
DDQN
online network argmax → selects a1 (online thinks a1 is best)
target network evaluates Q_target(s', a1) → 0.85 (more conservative)
Result: target = 0.85, closer to true Q value
Conclusion:
DDQN separates "who picks the action" from "who evaluates it" — like having a separate judge who doesn't know who nominated the candidate.

5 Identical Environment Logic

def step(self, action: int):
    svc = self.services[self._step]

    # ① violation detection (identical to DQNEnv)
    cap_violated      = bool(self.remaining_vms[action] < svc.requirement)
    conflict_violated = self._has_conflict(action, self._step)

    # ② compute penalty (identical to DQNEnv)
    cap_penalty      = -2.0 if cap_violated      else 0.0
    conflict_penalty = -2.0 if conflict_violated else 0.0
    ru = 0.0 if (cap_violated or conflict_violated) else \
         svc.requirement / (self.initial_vms[action] + 1e-8)

    # ③ force placement, remaining_vms may go negative (identical to DQNEnv)
    self.remaining_vms[action] -= svc.requirement
    self.ecu_placements[action].add(self._step)
    self._update_ecu_allowed(action, self._step)

    # ④ terminal bonus (identical to DQNEnv)
    done = self._step >= self.M
    terminal_bonus = 0.0
    if done:
        terminal_bonus = self.ar if total_viol == 0 else -self.ar

    return self._obs(), float(ru + cap_penalty + conflict_penalty + terminal_bonus), ...
    # ↑ return structure identical to DQNEnv.step()

NoneTraining Loop Pseudocode

DQN

for batch in replay_buffer:
    s, a, r, s_next = batch

    # target network selects + evaluates together
    a_best = argmax_a Q_target(s_next)
    y = r + γ * Q_target(s_next, a_best)

    loss = (Q_online(s, a) - y)²
    optimizer.step(loss)

DDQN

for batch in replay_buffer:
    s, a, r, s_next = batch

    # online selects, target evaluates
    a_best = argmax_a Q_online(s_next)
    y = r + γ * Q_target(s_next, a_best)

    loss = (Q_online(s, a) - y)²
    optimizer.step(loss)

Why is the AR gap small?

Comparison with PPO+ActionMask

In tight constraint scenarios, hard constraint enforcement (action masking) dominates soft constraint learning (reward penalties). DDQN's improvement over DQN is real but insufficient for constraint-critical applications.
