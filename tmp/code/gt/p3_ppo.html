1. What is a Gymnasium Environment?

  • reset()
  • step(action)
  • render()

Reinforcement Learning has two key players: the environment (the world) and the agent (the learner). Gymnasium defines a standard interface. The agent calls step() repeatedly to take actions and receive rewards, learning to maximize total reward over time.

类比 / Analogy: 把环境想象成一个棋盘游戏。reset() 是"重新摆棋"，step(action) 是"走一步"，然后裁判（环境）告诉你得了多少分、游戏是否结束。AI（智能体）通过不断练习学会下好棋。

P3Env specific task: each episode sequentially places M services onto N ECUs, one per step. The goal is to maximize final AR.

2. Class Header & Module Docstring

"""
P3 Environment — Unconstrained baseline (violations recorded, never penalized).

N < M: each ECU hosts multiple services.

Constraints:
    - Capacity violation → RECORDED only. Capacity redirect in step avoids
      violations when possible; forced-overflow fallback (all ECUs full) still
      places the service, allowing remaining_vms to go negative.
    - Conflict violation → RECORDED only. No penalty applied.

ru is always computed and added to _total_ru regardless of violations.
Terminal bonus: always +AR (unconstrained upper bound).
"""

class P3Env(gym.Env):
    """
    Each episode assigns M services to N ECUs (N < M), one service per step.
    Multiple services share ECUs.

    Observation (shape: 5N+6+M):
        [0]          current service demand (normalised)
        [1]          current cumulative AR
        [2]          sum of remaining ECU capacity (normalised, clipped ≥ 0)
        [3]          sum of remaining service demand (normalised)
        [4]          fraction of ECUs with sufficient capacity for current service
        [5]          fraction of services remaining
        [6:6+N]      initial capacity fraction per ECU
        [6+N:6+2N]   remaining capacity fraction per ECU
        [6+2N:6+3N]  conflict flag per ECU
        [6+3N:6+4N]  ECU allowed fraction
        [6+4N:6+5N]  valid-action flags (capacity only)
        [6+5N:6+5N+M] remaining service demands (sorted descending)
    """

The module docstring is the most important design decision: P3 records violations but never penalizes them. This makes P3 an unconstrained upper bound — useful as a theoretical ceiling to compare against constrained methods (P4, P5, etc.).

核心设计 / Core design: P3 = log violations (knows they happened) but no penalty (does not learn to avoid them). P3 = records violations (knows they happened) but never deducts reward (never learns to avoid them).

观测向量的形状是 5N + 6 + M，例如 N=7, M=10 时，形状为 5×7+6+10=51。 5 组 N 维的 ECU 特征 + 6 个全局标量 + M 个服务需求。

Observation shape: 5N + 6 + M. For N=7, M=10 that's 51 numbers. Five groups of per-ECU features (N each) + 6 global scalars + M service demands.

4. Conflict Helpers

_init_conflict_sets() — Generate Random Conflict Groups

def _init_conflict_sets(self, K: int = 10) -> list[set]:
    sets = []
    for _ in range(K):
        j = random.randint(2, self.M)
        sets.append(set(random.sample(range(self.M), j)))
    return sets

Generates K=10 random conflict groups. Each group has between 2 and M service indices, chosen randomly. Used only when no external scenario list is provided (standalone testing).

_has_conflict() — Check if an Assignment Would Conflict

def _has_conflict(self, ecu_idx: int, svc_idx: int) -> bool:
    return svc_idx not in self.ecu_allowed[ecu_idx]

ecu_allowed[j] is a set of service indices that can still be placed on ECU j without causing a conflict. If the current service is not in this set, placing it there would violate a conflict constraint.

_update_ecu_allowed() — Narrow Down Allowed Services After Placement

def _update_ecu_allowed(self, ecu_idx: int, svc_idx: int) -> None:
    for subset in self.conflict_sets:
        if svc_idx in subset:
            self.ecu_allowed[ecu_idx] -= (subset - {svc_idx})

After placing service svc_idx on ECU ecu_idx, we remove all conflicting peers from that ECU's allowed set. subset - {svc_idx} is set-difference: removes the placed service from the exclusion list (we exclude its peers, not itself).

类比 / Analogy: 小明（svc_idx）住进了宿舍 j。小明和小红、小华是"冲突组"（不能同住）。 于是从宿舍 j 的"可住人列表"中删去小红和小华。以后再给宿舍 j 分配时，小红和小华就不能进来了。

5. reset() — Starting a New Episode

def reset(self, seed=None, options=None):
    super().reset(seed=seed)
    if self._scenarios is not None:
        caps, reqs, _cs = random.choice(self._scenarios)
        self.ecus     = [ECU(f"ECU{i}", cap) for i, cap in enumerate(caps)]
        self.services = [SVC(f"SVC{i}", req) for i, req in enumerate(reqs)]
        self.initial_vms = np.array([e.capacity for e in self.ecus], dtype=np.float32)

        # Remap conflict-set indices from original order → sorted order
        sorted_order = sorted(range(len(self.services)),
                              key=lambda i: self.services[i].requirement, reverse=True)
        orig_to_new = {orig: new for new, orig in enumerate(sorted_order)}
        self.conflict_sets = [
            {orig_to_new[i] for i in cs if i < len(self.services)} for cs in _cs
        ]
    else:
        self.conflict_sets = self._init_conflict_sets()

    self.services = sorted(self.services, key=lambda s: s.requirement, reverse=True)
    self.remaining_vms   = self.initial_vms.copy()
    self.ecu_placements  = [set() for _ in range(self.N)]
    self.ecu_allowed     = [set(range(self.M)) for _ in range(self.N)]
    self.ar              = 0.0
    self._total_ru       = 0.0
    self._step           = 0
    self.capacity_violations  = 0
    self.conflict_violations  = 0
    self.valid_placed = 0
    self.episode_has_cap_violation      = False
    self.episode_has_conflict_violation = False
    return self._obs(), {}

If a scenarios list is provided, randomly pick one each episode. This forces the AI to generalize across different ECU/service configurations.

  • sorted_order
  • orig_to_new

Index remapping (tricky part): services are sorted by requirement descending, so their positions change. Conflict sets store original indices. We must remap them to match the new sorted order, or conflict checking would silently use wrong indices.

  • remaining_vms
  • ecu_placements
  • ecu_allowed
  • ar, _total_ru, _step
  • capacity_violations, conflict_violations
类比 / Analogy: reset() 就是"开始新的一局"——清空棋盘（ecu_placements=[]），重置分数（ar=0），重新洗牌（服务重新排序），准备好记分板（违规计数=0）。

6. action_masks() — Capacity-Only Mask

def action_masks(self) -> np.ndarray:
    if self._step >= self.M:
        return np.zeros(self.N, dtype=bool)
    svc = self.services[self._step]
    mask = (self.remaining_vms >= svc.requirement)
    if not np.any(mask):
        # Forced-overflow fallback: ECU with most remaining capacity.
        best = int(np.argmax(self.remaining_vms))
        mask = np.zeros(self.N, dtype=bool)
        mask[best] = True
    return mask
  • remaining_vms >= svc.requirement
  • 如果所有 ECU 都满了（mask 全 False）：选剩余容量最多的 ECU 作为"强制溢出"后备
  • 即使强制溢出，也不报错，只是 remaining_vms 会变成负数

P3 key distinction: only capacity is checked, conflicts are ignored entirely. If all ECUs are full, a "forced overflow" fallback picks the ECU with the most remaining capacity. remaining_vms can go negative — that's intentional in P3.

类比 / Analogy: 就像强行把第 10 位学生塞进一间只有 9 张床的宿舍——记录了"超员"，但不扣任何分，继续往下分配。
P3 vs P4 动作掩码核心区别：
P3: mask = (remaining_vms >= svc.requirement) — 只看容量
P4: mask = [(remaining_vms[j] >= svc.requirement) AND (not _has_conflict(j, svc_idx)) for j in range(N)] — 容量 AND 无冲突

7. _obs() — Building the Observation Vector

def _obs(self) -> np.ndarray:
    max_cap   = float(np.max(self.initial_vms) + 1e-8)
    total_cap = float(np.sum(self.initial_vms) + 1e-8)

    if self._step >= self.M:
        service_demand_norm          = np.float32(0.0)
        conflict_flag                = np.zeros(self.N, dtype=np.float32)
        valid_flag                   = np.zeros(self.N, dtype=np.float32)
        remaining_service_demand_sum = np.float32(0.0)
        remaining_services_count     = np.float32(0.0)
        remaining_usable_ecu_count   = np.float32(0.0)
    else:
        svc = self.services[self._step]
        service_demand_norm = np.float32(svc.requirement / max_cap)
        conflict_flag = np.array(
            [float(self._has_conflict(j, self._step)) for j in range(self.N)],
            dtype=np.float32,
        )
        valid_flag = (self.remaining_vms >= svc.requirement).astype(np.float32)
        remaining_service_demand_sum = np.float32(
            sum(self.services[t].requirement for t in range(self._step, self.M)) / total_cap
        )
        remaining_services_count   = np.float32((self.M - self._step) / max(self.M, 1))
        remaining_usable_ecu_count = np.float32(
            np.sum(self.remaining_vms >= svc.requirement) / max(self.N, 1)
        )

    remaining_abs_norm = np.clip(self.remaining_vms, -max_cap, max_cap) / max_cap
    initial_cap_pct    = self.initial_vms / max_cap
    remaining_usable_capacity_sum = np.float32(
        np.sum(np.clip(self.remaining_vms, 0.0, None)) / total_cap
    )
    remaining_svcs = np.array(
        [self.services[t].requirement / max_cap if t >= self._step else 0.0
         for t in range(self.M)],
        dtype=np.float32,
    )
    ecu_allowed_frac = np.array(
        [len(self.ecu_allowed[j]) / self.M for j in range(self.N)],
        dtype=np.float32,
    )

    return np.concatenate([
        [service_demand_norm],
        np.array([self.ar], dtype=np.float32),
        np.array([remaining_usable_capacity_sum], dtype=np.float32),
        np.array([remaining_service_demand_sum], dtype=np.float32),
        np.array([remaining_usable_ecu_count], dtype=np.float32),
        np.array([remaining_services_count], dtype=np.float32),
        initial_cap_pct,
        remaining_abs_norm,
        conflict_flag,
        ecu_allowed_frac,
        valid_flag,
        remaining_svcs,
    ]).astype(np.float32)
  • max_cap + 1e-8
  • remaining_abs_norm = clip(remaining_vms, -max_cap, max_cap) / max_cap
  • remaining_usable_capacity_sum = sum(clip(remaining_vms, 0, None)) / total_cap
  • valid_flag = (remaining_vms >= svc.requirement).astype(float32)
  • remaining_svcs[t] = 0.0 if t < _step else req_t / max_cap
  • ecu_allowed_frac = len(ecu_allowed[j]) / M

The conflict_flag array is computed and included in the observation (so the AI can see conflicts), but it is NOT used in the action mask. P3 gives the AI information about conflicts without enforcing them.

类比 / Analogy: 观测向量就像一份"仪表盘报告"交给 AI——告诉它每间宿舍还剩多少床位、现在要分配哪位学生、哪些宿舍住了"仇人"（冲突标志）、还有多少学生待分配……但 P3 只在门上贴了"快满了"的警告牌（容量掩码），没有贴"禁止入内"（冲突掩码）。

8. step() — The Core P3 Logic

def step(self, action: int):
    assert 0 <= action < self.N, f"Invalid action {action}"
    svc = self.services[self._step]

    cap_violated      = bool(self.remaining_vms[action] < svc.requirement)
    conflict_violated = self._has_conflict(action, self._step)

    if cap_violated:
        self.capacity_violations += 1
        self.episode_has_cap_violation = True
    if conflict_violated:
        self.conflict_violations += 1
        self.episode_has_conflict_violation = True
    if not (cap_violated or conflict_violated):
        self.valid_placed += 1

    # Violations are recorded only — no penalties applied in P3.
    ru = svc.requirement / (self.initial_vms[action] + 1e-8)

    self.remaining_vms[action] -= svc.requirement
    self.ecu_placements[action].add(self._step)
    self._update_ecu_allowed(action, self._step)
    self._total_ru += ru  # always add — unconstrained baseline
    _active = sum(1 for j in range(self.N) if self.ecu_placements[j])
    self.ar = self._total_ru / _active if _active > 0 else 0.0
    self._step += 1

    done = self._step >= self.M
    terminal_bonus = self.ar if done else 0.0  # always +AR

    total_viol = self.capacity_violations + self.conflict_violations
    reward = float(ru + terminal_bonus)
    info = {
        "ar":                  self.ar,
        "capacity_violations": self.capacity_violations,
        "conflict_violations": self.conflict_violations,
        # ... more keys ...
    }
    return self._obs(), reward, done, False, info

Violations are detected and counted but do NOT affect the reward. The AI never receives negative feedback for violating constraints. 

ru = svc.requirement / (self.initial_vms[action] + 1e-8)
self._total_ru += ru  # always add — unconstrained baseline

ru = req / initial_cap — always positive, always added to the total, regardless of violations. Even if remaining_vms goes negative (capacity exceeded), ru is still counted as if the placement was valid.

关键3：terminal_bonus 总是 +AR

terminal_bonus = self.ar if done else 0.0  # always +AR
reward = float(ru + terminal_bonus)

At the final step, the AI receives a bonus equal to the full AR value. Since AR is never reduced by violations, this creates a strong incentive to maximize AR even at the cost of unlimited violations.

When remaining_vms goes negative, more VMs are "used" than the ECU has, making per-ECU utilization > 1.0. AR = total_util / active_ECUs can therefore exceed 1.0 in P3. This is intentional — P3 represents the unconstrained ceiling.

类比 / Analogy: P3 就像一个"无限超员模式"的宿舍分配——可以把 100 人塞进 10 张床的宿舍里， "利用率"计算时会显示 1000%，虽然现实中不可能，但作为理论上限很有参考价值。 P3 的目标是告诉我们：如果没有任何约束，最高能达到多少 AR？

9. P3 vs P4 — Key Differences Comparison Card

P4 is the "hard-masked" version — capacity AND conflict constraints are both enforced via action masking. Below is a side-by-side comparison of the key design differences.

P3 — Unconstrained Baseline

violations recorded, never penalized

# action_masks(): capacity only
mask = (self.remaining_vms >= svc.requirement)
# conflict NOT checked

# step(): ru always counted
ru = svc.requirement / (initial_vms[action] + 1e-8)
self._total_ru += ru  # unconditional

# terminal bonus: always +AR
terminal_bonus = self.ar if done else 0.0

reward = ru + terminal_bonus
# no violation penalty ever

Result: AR can exceed 1.0 (remaining_vms goes negative). AI never learns to respect constraints.

结果：AR 可超过 1.0（remaining_vms 变负数）。AI 永远不会学习遵守约束。

P4 — Hard Capacity + Conflict Mask

both constraints enforced; violations only from forced-overflow fallback

# action_masks(): capacity AND conflict
mask = np.array([
    (remaining_vms[j] >= svc.requirement)
    and (not _has_conflict(j, svc_idx))
    for j in range(N)], dtype=bool)

# step(): ru = 0 if violated
violated = cap_violated or conflict_violated
violation_penalty = -2.0 if violated else 0.0
ru = 0.0 if violated else svc.requirement / initial_vms[action]
if ru > 0: self._total_ru += ru  # only valid

# terminal bonus: reduced if any violations
if done:
    terminal_bonus = ar if total_viol == 0 else 0.1 * ar

reward = ru + violation_penalty + terminal_bonus

Result: AI learns to avoid violations. AR ≤ 1.0 always. Forced-overflow (only when no valid ECU exists) still incurs -2.0.

结果：AI 学会避免违规。AR 始终 ≤ 1.0。只有在所有 ECU 都满时才有强制溢出（-2.0 惩罚）。

Feature P3 P4
Capacity ）Only in mask Mask + penalty
Conflict Never enforced Mask + penalty
ru Always counted 0 Set to 0
Penalty None -2.0 Heavy penalty
Terminal bonus +AR Always +AR +0.1×AR
> 1? Yes (unconstrained) No (constrained)
Purpose Upper bound ceiling Feasible solution

Conclusion: P3's AR is a theoretical ceiling. When comparing constrained methods (P4, P5, etc.), we can compute how close they get to P3's AR as a measure of efficiency. A constrained AR / P3 AR ratio close to 1.0 means the constraints have little cost.
