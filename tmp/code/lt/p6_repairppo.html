— The core innovation of P6
"""
P6 Environment — PPO + Best-fit Repair Heuristic.

N < M: each ECU hosts multiple services.

Constraints:
    - Capacity or conflict violation → best-fit repair heuristic relocates the
      service to the valid ECU with the highest utilisation density (req/cap).
    - A repaired step incurs a small penalty (-0.1) to signal suboptimal choice.
    - If no valid ECU exists for repair → episode terminates with demand penalty.
    - Terminal bonus: +AR * (1 - repair_rate), rewarding episodes with fewer repairs.
"""

Chinese Explanation

English Explanation

Compare the three strategies across P4/P5/P6:

  • P4 (action masking): illegal actions are blocked before the agent ever picks them — the agent can only see valid choices
  • P5/DQN: agent picks freely, but illegal choices give a large negative reward
  • P6 (repair): agent picks freely, but when it picks an invalid ECU, the environment silently fixes it to the best valid ECU. A small -0.1 penalty signals the mistake without crashing the episode

This lets the agent learn from mistakes softly, while still completing every episode productively.

— the heart of P6's repair strategy
def _best_fit_repair(self, svc_idx: int) -> int | None:
    """Return ECU index with highest req/cap among capacity-valid, conflict-free ECUs."""
    svc = self.services[svc_idx]

    # Step 1: collect ALL ECUs that satisfy BOTH conditions
    valid = [
        j for j in range(self.N)
        if self.remaining_vms[j] >= svc.requirement     # ← capacity OK
        and svc_idx in self.ecu_allowed[j]             # ← no conflict
    ]

    # Step 2: if no valid ECU found → return None (triggers PATH C)
    if not valid:
        return None

    # Step 3: among valid ECUs, pick the one with highest utilization density
    return max(valid, key=lambda j: svc.requirement / (self.initial_vms[j] + 1e-8))
  • ECU
  • ECU → returns None, causing step() to trigger "Path C" (episode terminated)

Why req / initial_cap? (Best-fit intuition)

Think of it like packing boxes: if a service needs 80 units and ECU-A has 100 capacity, it fills 80% of ECU-A. If ECU-B has 200 capacity, it only fills 40%. The "best-fit" heuristic prefers ECU-A because it wastes less space — smaller ECUs get used more efficiently. This mirrors classic bin-packing best-fit algorithms.

Note: 1e-8 is added to avoid division by zero if initial capacity is somehow 0.

def step(self, action: int):
    svc = self.services[self._step]

    # ── Detect violations ──
    cap_violated      = bool(self.remaining_vms[action] < svc.requirement)
    conflict_violated = self._has_conflict(action, self._step)

    was_repaired = False
    if cap_violated or conflict_violated:
        repaired = self._best_fit_repair(self._step)

        if repaired is None:           # ── PATH C: repair failed ──
            unplaced_demand = sum(self.services[i].requirement
                                  for i in range(self._step, self.M))
            penalty = -float(unplaced_demand) / (float(np.sum(self.initial_vms)) + 1e-8)
            return self._obs(), penalty, True, False, {...}

        action = repaired              # ── PATH B: use repaired ECU ──
        was_repaired = True
        self.repairs += 1
        if cap_violated:
            self.cap_violations += 1
        if conflict_violated:
            self.conflict_violations += 1
    # else: PATH A — agent chose correctly ──

    # ── Compute reward ──
    repair_penalty = -0.1 if was_repaired else 0.0
    ar_prev = self.ar  # capture before placement for Δar
    ru = svc.requirement / (self.initial_vms[action] + 1e-8)
    # ru = utilization density; accumulated to compute AR

    self.remaining_vms[action] -= svc.requirement   # deduct capacity
    _was_empty = not self.ecu_placements[action]
    self.ecu_placements[action].add(self._step)      # record deployment
    if _was_empty:
        self._n_active += 1
    self._update_ecu_allowed(action, self._step)     # update conflict sets
    self._total_ru += ru
    _active = self._n_active
    self.ar = self._total_ru / _active
    self._step += 1
    self.valid_placed += 1

    delta_ar = self.ar - ar_prev  # potential-based shaped reward
    return self._obs(), float(delta_ar + repair_penalty + terminal_bonus), done, False, info

What is Δar?

Δar = ar_new - ar_prev

  • ru = svc.requirement / (initial_vms[action] + 1e-8) (utilisation density)
  • accumulate ru each step → AR = _total_ru / active ECU count
  • step reward = Δar (AR increment), not ru directly
  • Δar is potential-based reward shaping: dense signal that preserves optimal policy

# Total step reward:
return ..., float(ru + repair_penalty + terminal_bonus), ...

terminal_bonus = AR × (1 - repair_rate)：

  • repair_rate = 0 (no repairs) → bonus = AR × 1.0 = full bonus
  • repair_rate = 0.5 (half repaired) → bonus = AR × 0.5 = half bonus
  • repair_rate = 1.0 (every step repaired) → bonus = AR × 0 = no terminal bonus

English: Why multiply by AR?

The terminal bonus is scaled by the current AR (average utilization). This means:

  • An episode that packs services efficiently (high AR) AND makes few mistakes earns the largest bonus
  • An episode with many repairs earns a smaller fraction of its own AR score
  • This creates a compound incentive: the agent wants both high AR and low repair rate simultaneously
Terminal Bonus Examples (AR = 0.6)
0 repairs / 10 steps → rate=0.0 → bonus = 0.6 × 1.0 = +0.60
3 repairs / 10 steps → rate=0.3 → bonus = 0.6 × 0.7 = +0.42
5 repairs / 10 steps → rate=0.5 → bonus = 0.6 × 0.5 = +0.30
10 repairs / 10 steps → rate=1.0 → bonus = 0.6 × 0.0 = +0.00

def reset(self, seed=None, options=None):
    super().reset(seed=seed)
    if self._scenarios is not None:
        caps, reqs, _cs = random.choice(self._scenarios)
        self.ecus     = [ECU(f"ECU{i}", cap) for i, cap in enumerate(caps)]
        self.services = [SVC(f"SVC{i}", req) for i, req in enumerate(reqs)]
        self.initial_vms = np.array([e.capacity for e in self.ecus], dtype=np.float32)
        self.conflict_sets = [set(s) for s in _cs]
    else:
        self.conflict_sets = self._init_conflict_sets()

    # Sort services descending by requirement (FFD order).
    # Remap conflict_set indices so self._step always indexes sorted services.
    sort_idx = sorted(range(self.M), key=lambda i: -self.services[i].requirement)
    self.services = [self.services[i] for i in sort_idx]
    inv_perm = [0] * self.M
    for new_i, old_i in enumerate(sort_idx):
        inv_perm[old_i] = new_i
    self.conflict_sets = [{inv_perm[k] for k in cs} for cs in self.conflict_sets]
    self.remaining_vms   = self.initial_vms.copy()
    self.ecu_placements  = [set() for _ in range(self.N)]
    self.ecu_allowed     = [set(range(self.M)) for _ in range(self.N)]

    # reset all counters
    self.ar              = 0.0
    self._step           = 0
    self.repairs             = 0   # ← P6-specific
    self.cap_violations      = 0   # ← P6-specific
    self.conflict_violations = 0   # ← P6-specific
    return self._obs(), {}
  • （FFD）
  • （sort_idx/inv_perm）
  • ecu_allowed[j] = set(range(M))
  • All P6-specific counters (repairs, cap_violations, conflict_violations) reset to zero

Why sort by requirement descending?

Greedy scheduling insight: large services are harder to fit (they consume more capacity). If we place large services first, we ensure they get placed while there's still ample capacity. Small services are more flexible and can fill the remaining gaps. Placing small ones first risks leaving no room for large ones — leading to PATH C early terminations.
