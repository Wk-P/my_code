设计思想 / Design Philosophy (Lines 1–17)

P7 的独特特性 — P7's unique feature: joint action space
"""
P7 Environment — Joint SVC-selection + ECU-assignment (Sequence-to-Sequence ordering).

Unlike P4 which processes SVCs in a fixed sorted order, P7 lets the agent freely
choose WHICH SVC to place at each step AND which ECU to assign it to.
This avoids greedy dead-ends by learning optimal placement ordering.

Action space : Discrete(M × N)
    action = svc_idx * N + ecu_idx
    svc_idx : which unplaced SVC to assign  (0..M-1)
    ecu_idx : which ECU to assign it to     (0..N-1)

Observation (shape 4 + 4N + 3M):
    Global  (4)   : AR, remaining_cap/total_cap, remaining_demand/total_cap, frac_svcs_remaining
    Per-ECU (4N)  : init_cap/max_cap, rem_cap/max_cap, ecu_allowed_frac, svcs_on_ecu/M
    Per-SVC (3M)  : req/max_cap, placed_flag, n_valid_ecus/N
"""

中文解读 — P7 的核心差异

P4 到 P6 都是"固定顺序"——服务按需求降序排列，AI 只需要每步选哪个 ECU。

P7 的革命性改变：AI 每步要同时决定两件事：

  • 选哪个 服务（SVC） 来部署（0 到 M-1）
  • 把它分配到哪个 ECU（0 到 N-1）

这两个选择被编码成一个数字：action = svc_idx × N + ecu_idx

动作空间大小 = M × N（例如 M=15, N=10 → 150 个可能动作）

为什么要让 AI 决定顺序? / Why let AI choose order?

固定排序（P4/P6）的问题："需求量大→先放"的贪心策略有时会导致死局。例如：如果大服务 A 和大服务 B 只能共用同一个 ECU，先放 A 就导致 B 无路可放。

P7 的设想：让 AI 学会最优的放置顺序，避免贪心策略带来的死局。理论上 P7 可以找到比 P4 更优的解。

The fixed greedy order in P4 can create "dead-ends." P7 hopes the agent learns smarter orderings — for instance, placing a constrained service first before resources are consumed.

代价 / The Cost

P7 的动作空间是 P4 的 M 倍大（150 vs 10 for typical settings）。更大的搜索空间 → 更难学习，尤其是在 5M 步有限训练预算下。实验中 P7 往往在 LT 测试场景下表现欠佳。

def __init__(self, ecus: list[ECU], services: list[SVC], scenarios=None):
    super().__init__()
    self.N = len(ecus)         # 10 ECUs
    self.M = len(services)     # 15 services

    # ── 联合动作空间 / Joint action space ──
    self.action_space = gym.spaces.Discrete(self.M * self.N)
    # P4: Discrete(N=10)     → 10 possible actions per step
    # P7: Discrete(M×N=150)  → 150 possible actions per step (15x larger!)

    # ── 观测空间 / Observation space ──
    self.observation_space = gym.spaces.Box(
        low=-1.0, high=1.0,
        shape=(4 + 4 * self.N + 3 * self.M,),  # = 4 + 40 + 45 = 89 dims
        dtype=np.float32,
    )

    self.initial_vms = np.array([e.capacity for e in ecus], dtype=np.float32)
    self.reset()

action_masks() — 动态掩码 / Dynamic Action Mask

M×N 位掩码，精确屏蔽无效的 (SVC, ECU) 组合
def action_masks(self) -> np.ndarray:
    mask = np.zeros(self.M * self.N, dtype=bool)
    # mask[i*N + j] corresponds to: "place SVC i on ECU j"
    has_valid = False

    for i in range(self.M):                          # iterate over all SVCs
        if self.placed[i]:
            continue                                    # skip already-placed SVCs
        svc_req = self.services[i].requirement
        for j in range(self.N):                        # iterate over all ECUs
            if (self.remaining_vms[j] >= svc_req)       # capacity check
            and (not self._has_conflict(j, i)):        # conflict check
                mask[i * self.N + j] = True
                has_valid = True

    if has_valid:
        return mask

    # ── Fallback: no valid (SVC, ECU) pair exists ──
    # Force each remaining SVC to the ECU with most capacity
    best_j = int(np.argmax(self.remaining_vms))
    for i in range(self.M):
        if not self.placed[i]:
            mask[i * self.N + best_j] = True
    return mask

中文解读

掩码数组有 M×N 位（例如 150 位）。每一位对应一个 (SVC, ECU) 组合：

  • mask[i × N + j] = True：允许把服务 i 放到 ECU j
  • mask[i × N + j] = False：禁止这个组合（已放置/容量不足/有冲突）

三种情况会让 mask 位为 False：

  1. self.placed[i] == True：服务 i 已经部署过了
  2. remaining_vms[j] < svc_req：ECU j 的剩余容量不够
  3. _has_conflict(j, i) == True：ECU j 不允许放服务 i（冲突约束）

Fallback 兜底机制 / Fallback Mechanism

如果遍历完所有 (SVC, ECU) 组合后 has_valid == False（没有任何合法动作），那么：

  • 找出剩余容量最多的 ECU（np.argmax(remaining_vms)）
  • 让每个未部署的服务都可以选这个 ECU（即使可能违规）
  • 这样保证掩码至少有一位为 True，让训练不会卡住

这是"软强制"兜底——步入绝境时至少让 AI 做点什么，而不是抛出错误。

Mask Example (M=4 SVCs, N=3 ECUs, 12 bits total)
ECU0 ECU1 ECU2
SVC0 (placed) → [False] [False] [False] ← already placed, all blocked
SVC1 → [True] [False] [True] ← ECU1 has no capacity
SVC2 → [False] [True] [False] ← ECU0,ECU2 conflict
SVC3 → [True] [True] [True] ← all valid

_obs() — 三组观测特征 / Three Feature Groups

4 + 4N + 3M 维观测向量的构建
def _obs(self) -> np.ndarray:
    max_cap   = float(np.max(self.initial_vms) + 1e-8)
    total_cap = float(np.sum(self.initial_vms) + 1e-8)

    # ──── Group 1: Global features (4 values) ────────────────────────
    remaining_cap_sum = float(np.sum(np.clip(self.remaining_vms, 0, None))) / total_cap
    remaining_dem_sum = float(
        sum(self.services[i].requirement for i in range(self.M) if not self.placed[i])
    ) / total_cap
    frac_remaining = float(np.sum(~self.placed)) / max(self.M, 1)
    # → [self.ar, remaining_cap_sum, remaining_dem_sum, frac_remaining]

    # ──── Group 2: Per-ECU features (4N values) ──────────────────────
    init_cap_frac    = self.initial_vms / max_cap        # 初始容量占比
    rem_cap_frac     = np.clip(self.remaining_vms, -max_cap, max_cap) / max_cap  # 剩余容量占比
    ecu_allowed_frac = np.array(
        [len(self.ecu_allowed[j]) / self.M for j in range(self.N)], dtype=np.float32
    )  # 该 ECU 还能放多少服务（比例）
    svcs_on_ecu_frac = np.array(
        [len(self.ecu_placements[j]) / self.M for j in range(self.N)], dtype=np.float32
    )  # 已放了多少服务在该 ECU 上（比例）

    # ──── Group 3: Per-SVC features (3M values) ──────────────────────
    req_norm    = np.array([s.requirement / max_cap for s in self.services], dtype=np.float32)
    placed_flag = self.placed.astype(np.float32)           # 0.0 or 1.0
    valid_ecus_frac = np.array([
        (sum(1 for j in range(self.N)
             if (self.remaining_vms[j] >= self.services[i].requirement)
             and (not self._has_conflict(j, i))
        ) / self.N) if not self.placed[i] else 0.0
        for i in range(self.M)
    ], dtype=np.float32)  # 该服务有多少个合法 ECU 可选（比例）

    return np.concatenate([
        [self.ar, remaining_cap_sum, remaining_dem_sum, frac_remaining],  # 4
        init_cap_frac, rem_cap_frac, ecu_allowed_frac, svcs_on_ecu_frac,  # 4N
        req_norm, placed_flag, valid_ecus_frac,                            # 3M
    ]).astype(np.float32)

为什么 Per-SVC 比 P6 多了信息？/ Why more SVC info than P6?

P6 只需要知道"当前服务"（固定顺序，每步处理同一个），所以 SVC 信息只需标量。P7 因为 AI 自由选择哪个服务，所以每步需要知道所有 M 个服务的状态（哪些已放完、哪些还有合法 ECU、需求多大），才能做出明智的选择。

reset() — 回合重置 / Episode Reset

P7 不排序服务 — services keep original YAML order
def reset(self, seed=None, options=None):
    super().reset(seed=seed)
    if self._scenarios is not None:
        caps, reqs, _cs = random.choice(self._scenarios)
        self.ecus     = [ECU(f"ECU{i}", cap) for i, cap in enumerate(caps)]
        self.services = [SVC(f"SVC{i}", req) for i, req in enumerate(reqs)]
        self.initial_vms = np.array([e.capacity for e in self.ecus], dtype=np.float32)
        # ── 关键差异：不重映射冲突集合索引！保持原始 YAML 顺序 ──
        self.conflict_sets = [set(cs) for cs in _cs]   # No index remapping!
    else:
        self.conflict_sets = self._init_conflict_sets()

    self.N = len(self.ecus)
    self.M = len(self.services)
    self.remaining_vms  = self.initial_vms.copy()
    self.ecu_placements = [set() for _ in range(self.N)]
    self.ecu_allowed    = [set(range(self.M)) for _ in range(self.N)]

    # ── P7-specific: track which services have been placed ──
    self.placed         = np.zeros(self.M, dtype=bool)
    # placed[i] = False → service i is still unplaced
    # placed[i] = True  → service i has been assigned to an ECU

    self.ar             = 0.0
    self._total_ru      = 0.0
    self._step          = 0
    self.capacity_violations = 0
    self.conflict_violations = 0
    return self._obs(), {}

step() — 解码动作并执行 / Decode and Execute

联合动作的分解与奖励计算
def step(self, action: int):
    # ── Step 1: 解码联合动作 / Decode joint action ──
    svc_idx = int(action) // self.N    # which SVC to place
    ecu_idx = int(action) %  self.N    # which ECU to assign it to

    assert not self.placed[svc_idx], f"SVC {svc_idx} already placed!"
    svc = self.services[svc_idx]

    # ── Step 2: 检测违规 / Check violations ──
    cap_violated      = bool(self.remaining_vms[ecu_idx] < svc.requirement)
    conflict_violated = self._has_conflict(ecu_idx, svc_idx)

    if cap_violated:
        self.capacity_violations += 1
        self.episode_has_cap_violation = True
    if conflict_violated:
        self.conflict_violations += 1
        self.episode_has_conflict_violation = True
    if not (cap_violated or conflict_violated):
        self.valid_placed += 1

    # ── Step 3: 计算奖励 / Compute reward ──
    violated          = cap_violated or conflict_violated
    violation_penalty = -2.0 if violated else 0.0
    ru = 0.0 if violated else svc.requirement / (self.initial_vms[ecu_idx] + 1e-8)
    # Note: violated step gets ru=0 AND penalty=-2.0 (double punishment)

    # ── Step 4: 更新状态 / Update state ──
    self.remaining_vms[ecu_idx] -= svc.requirement
    self.ecu_placements[ecu_idx].add(svc_idx)
    self._update_ecu_allowed(ecu_idx, svc_idx)
    self.placed[svc_idx] = True           # ← mark as placed
    self._step += 1

    # ── Step 5: 检查是否完成 / Check done ──
    done       = self._step >= self.M     # all M services placed
    total_viol = self.capacity_violations + self.conflict_violations
    terminal_bonus = 0.0
    if done:
        terminal_bonus = self.ar if total_viol == 0 else 0.1 * self.ar

    return self._obs(), float(ru + violation_penalty + terminal_bonus), done, False, info

Terminal Bonus: P6 vs P7 Comparison

策略零违规时 terminal bonus有违规时 terminal bonus
P6 (repair) AR × (1 − 0) = AR AR × (1 − repair_rate)（线性递减）
P7 (joint) AR × 1.0 = AR（满额） AR × 0.1（直接打九折，陡降）

P7 用更严格的二元惩罚机制：有任何违规 → terminal bonus 直接变 0.1×AR（而非线性递减）。这给 AI 一个强烈的信号：任何违规都是严重的。

为什么违规步骤 ru=0？/ Why ru=0 for violated steps?

ru = 0.0 if violated else svc.requirement / (initial_vms[ecu_idx] + 1e-8)

违规时：不仅 penalty=-2.0，还得不到利用率奖励（ru=0）。这是双重惩罚：

  • 没有正向奖励 → 学不到好的利用率
  • 还有 -2.0 大惩罚 → 明确信号"这是错误的"

注意：即使违规，服务仍然被部署（placed[svc_idx] = True），状态仍然更新。P7 不像 P6 那样"修复"——它让错误发生，但给予大惩罚来引导学习。

为什么 P7 在 LT 场景下表现欠佳 / Why P7 Struggles in LT

动作空间过大带来的学习困难

核心问题：动作空间维度爆炸 / The Curse of Action Space Size

P7 的动作空间是 M × N = 150（P4 是 10，P7 是 P4 的 15 倍大！）

在稀缺资源环境下（15 个服务，10 个 ECU），150 个可能动作中：

  • 已部署服务对应的动作：被掩码屏蔽（约占 M 个服务逐步增加）
  • 容量不足的 (SVC, ECU) 对：被掩码屏蔽
  • 有冲突的 (SVC, ECU) 对：被掩码屏蔽
  • 实际有效动作可能只有 5-20 个，剩余 130+ 个都无效

AI 需要在 5M 训练步骤内从一个高维、稀疏有效动作的空间中学习，这比 P4 的 10 维小空间困难得多。

Action Space Comparison
P4 (fixed order): action ∈ {0..9} → 10 choices per step
only ~3-7 valid (masked)

P6 (repair): action ∈ {0..9} → 10 choices per step
all 10 allowed (repair handles violations)

P7 (joint): action ∈ {0..149} → 150 choices per step
typically only 5-20 valid (sparse!)

With 5M training steps, P7 explores far fewer unique (state, action) pairs.
The signal is diluted across a 15x larger space — learning is much harder.

LT（长尾）场景特别难 / Why LT Makes It Worse

  • LT = 稀缺资源：ECU 容量少，服务需求高，合法位置更少
  • 合法动作更稀疏 → AI 更难遇到有效的学习信号
  • 随机探索 150 个动作 → 大概率选到违规动作 → 大量 -2.0 惩罚 → 梯度信号嘈杂
  • 即使 action_masks 屏蔽了大多数无效动作，剩余有效动作的分布也随时间剧烈变化（随着 placed 更新）

相比之下，P4 每步只需要从 10 个动作中选，其中 mask 后只有少量选项，学习效率高得多。

理论 vs 实践 / Theory vs Practice

理论上：P7 应该更强——它能学到最优的服务放置顺序，避免贪心死局。

实践中：给定 5M 步预算，P7 的庞大动作空间导致样本效率太低，往往没有完全收敛。P4 的小动作空间 + 排序启发式往往在 5M 步内就能达到接近最优。

这体现了强化学习中的基本权衡：表达力 vs 可学习性（Expressiveness vs Learnability）。
