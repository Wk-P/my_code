2. What Problem Are We Solving?

We have N ECUs (fewer, e.g. 5) and M services (more, e.g. 12). Since N < M, each ECU must host multiple services. 

AR = (1 / active_ECUs) × Σ_j [ Σ_{i→j} req_i / cap_j ]
   = average utilisation per active ECU

AR is a fractional objective — numerator is total utilization, denominator is the number of active ECUs (which depends on the solution itself). We use Dinkelbach's algorithm to handle this.

Constraint Meaning
Assignment Each service goes to exactly one ECU
容量 Capacity — An ECU's total demand ≤ its capacity
冲突 Conflict ECU — Conflicting services must be separated

3. solve_service_deployment() — Function Signature & Docstring

def solve_service_deployment(M, N, e_list, n_list, conflict_sets=None, seed=None):
    """
    Maximise AR = total_util / active_ecus via Dinkelbach's algorithm.

    AR is a fractional objective (denominator active_ecus depends on the solution),
    so we cannot optimise it directly with a single ILP.  Dinkelbach's method
    iteratively solves the parametric sub-problem:
        max  F(x,y) - λ * G(y)
    where F = Σ x[i][j]*req_i/cap_j  and  G = Σ y_j (active ECUs),
    updating λ ← F/G at each step until |Δλ| < tol.

    Parameters:
    - M: number of services
    - N: number of ECUs  (N < M; multiple services share ECUs)
    - e_list: list of ECU capacities (length N)
    - n_list: list of service requirements (length M)
    - conflict_sets: list of sets of service indices
    - seed: random seed (optional)

    Returns:
    - dict with keys: status, total_utilization, active_ecus,
                      avg_utilization, allocation
    """
  • M
  • N
  • e_list
  • n_list
  • conflict_sets
  • seed

Return value: a dictionary with the final assignment, active ECU count, and AR score.

类比 / Analogy: 这个函数就像一个"聪明的宿舍分配员"。你告诉它有多少间宿舍（N）、多少学生（M）、每间有多少床位（e_list）、每个学生需要多少床位（n_list）、哪些学生不能住同一间（conflict_sets），它会给出最优分配方案。

4. Dinkelbach Outer Loop

lambda_val = 0.0
tol        = 1e-6
max_iter   = 20
best_x = best_y = None

for iteration in range(max_iter):
    prob = pulp.LpProblem(f"Maximize_AR_iter{iteration}", pulp.LpMaximize)

λ (lambda_val) is the key parameter — it starts at 0 and gradually converges to the true AR maximum. Each iteration solves a fresh ILP with the current λ baked into the objective.

max_iter = 20: cap at 20 iterations, but in practice 3–5 are enough. A fresh LpProblem is created each iteration because the objective changes with λ.

类比 / Analogy: 就像你去黑板前猜答案，猜完之后看看实际结果，再回去改。λ 就是每次的"猜测值"，不断逼近真实答案。

5. Decision Variables — x[i][j] and y[j]

# x[i][j] = 1 if service i assigned to ECU j
x = pulp.LpVariable.dicts("x", (range(M), range(N)), cat='Binary')

# y[j] = 1 if ECU j has at least one service (active)
y = pulp.LpVariable.dicts("y", range(N), cat='Binary')

x[i][j] — a binary (0 or 1) decision variable. 1 means "service i is placed on ECU j", 0 means it isn't. There are M×N such variables total.

y[j] — 1 if ECU j is "active" (hosts at least one service), 0 otherwise. N variables total, used to count active ECUs (the denominator of AR).

类比 / Analogy: 把宿舍分配写在一张表格上：x[i][j] 就是第 i 个学生是否住在第 j 间宿舍（打勾/不打勾）。y[j] 就是第 j 间宿舍是否有人（开灯/关灯）。

6. Objective Function — F(x) − λ·G(y)

# Parametric objective: F(x) - λ * G(y)
prob += (
    pulp.lpSum(x[i][j] * n_list[i] / e_list[j]
               for i in range(M) for j in range(N))
    - lambda_val * pulp.lpSum(y[j] for j in range(N))
)
  • F(x) Σ x[i][j] × req_i / cap_j — 所有已分配服务对其所在 ECU 的利用率之和。 Sum of (service demand / ECU capacity) for every assigned pair.
  • G(y) Σ y[j] — 活跃 ECU 的数量。 Count of active ECUs.

The full objective is F(x) − λ × G(y). Dinkelbach's theorem guarantees: when the optimal value of this parametric problem equals 0, λ equals the true maximum of AR = F/G.

类比 / Analogy: 想象 λ 是"开灯成本"。每点亮一间宿舍（y[j]=1）就要付 λ 元。解题器会在"提高利用率收益"和"少开房间节省成本"之间找平衡点。当 λ 等于真实 AR 时，两者完美平衡，我们找到了最优解。

7. Five Constraints

NoneConstraint 1 — Assignment (

# Each service assigned to exactly one ECU
for i in range(M):
    prob += pulp.lpSum(x[i][j] for j in range(N)) == 1, f"Assign_{i}"

For each service i, exactly one x[i][j] can be 1. This ensures every service is deployed to exactly one ECU — no service is left out, none is deployed twice.

类比: 每位学生必须且只能住在一间宿舍。不能无家可归，也不能同时住两间。

NoneConstraint 2 — Capacity (

# Capacity constraint per ECU
for j in range(N):
    prob += pulp.lpSum(x[i][j] * n_list[i] for i in range(M)) <= e_list[j], f"Cap_{j}"

For each ECU j, the total VM demand of all services placed there must not exceed its capacity. When x[i][j]=0 the product is 0, so only assigned services count.

类比: 宿舍不能超过床位数——住进去的人的床位需求总和 ≤ 宿舍床位总数。

NoneConstraint 3 & 4 — Linking LB / UB (

# Linking: x[i][j]=1 forces y[j]=1; no service means y[j] must be 0
for j in range(N):
    prob += pulp.lpSum(x[i][j] for i in range(M)) >= y[j], f"LinkLB_{j}"
    for i in range(M):
        prob += x[i][j] <= y[j], f"LinkUB_{i}_{j}"
  • LinkLB Σ x[i][j] ≥ y[j]
  • LinkUB x[i][j] ≤ y[j]：如果服务 i 放在 ECU j（x[i][j]=1），则 y[j] 也必须是 1。（你不能把人放在一间"关灯"的宿舍）

These two constraints ensure y[j] and x[i][j] are consistent: y[j]=1 iff at least one service is assigned to ECU j.

类比: 宿舍有人住（y=1）当且仅当至少有一位学生住进去（x=1）。不允许"空宿舍算开灯"，也不允许"有人住但算关灯"。

NoneConstraint 5 — Hard Infeasibility Pre-filter (

# Hard-infeasible assignments
for i in range(M):
    for j in range(N):
        if n_list[i] > e_list[j]:
            prob += x[i][j] == 0, f"Infeasible_{i}_{j}"

If a service requires more VMs than an ECU can hold, we pre-set that x[i][j]=0, saving the solver from wasting time on impossible assignments.

NoneConstraint 6 — Conflict (

# Conflict constraints: at most one service from each conflict subset per ECU
if conflict_sets:
    for k, subset in enumerate(conflict_sets):
        valid = [i for i in subset if i < M]
        if len(valid) >= 2:
            for j in range(N):
                prob += pulp.lpSum(x[i][j] for i in valid) <= 1, f"Conflict_{k}_ECU_{j}"

Conflict sets define groups of mutually incompatible services — at most one service from each conflict group may be placed on any single ECU. This models resource conflicts (e.g., two services contend for the same hardware peripheral).

类比: 某些学生互相吵架，不能住同一间宿舍。冲突约束就是"吵架组"规则：同一组里最多只能有一人住在同一间房。

8. λ Update and Convergence

# Solve this iteration's ILP
prob.solve(pulp.PULP_CBC_CMD(msg=False))

if pulp.LpStatus[prob.status] != 'Optimal':
    break

best_x, best_y = x, y

active = sum(1 for j in range(N)
             if pulp.value(y[j]) is not None and pulp.value(y[j]) > 0.5)
total  = sum(
    pulp.value(x[i][j]) * n_list[i] / e_list[j]
    for i in range(M) for j in range(N)
    if pulp.value(x[i][j]) is not None and pulp.value(x[i][j]) > 0.5
)

new_lambda = total / active
if abs(new_lambda - lambda_val) < tol:
    lambda_val = new_lambda
    break
lambda_val = new_lambda
  • prob.solve(PULP_CBC_CMD(msg=False))
  • LpStatus[prob.status] != 'Optimal'
  • best_x, best_y = x, y
  • active = sum(1 for j ... if value(y[j]) > 0.5)
  • total = sum(...)
  • new_lambda = total / active
  • if abs(new_lambda - lambda_val) < tol: break

After each solve, compute the true AR of the current solution and use it as the new λ. When λ stops changing (converges), we have found the globally optimal AR.

类比 / Analogy: 就像反复猜"这次分配的平均利用率是多少"，然后拿猜测结果去重新优化，再看结果，直到两次结果完全一样（误差 < 0.000001）。

数学保证（Dinkelbach定理）：每次迭代后，AR 单调增加，且最终收敛到全局最优解。
Mathematical guarantee (Dinkelbach): AR strictly increases with each iteration and converges to the global optimum.

9. main() — Loading YAML & Caching Results

@timer_utils.timer
def main(**kwargs):
    # read config from YAML file
    config_path = kwargs.get('config_path', None)

    # If no path provided, find the latest config file
    if config_path is None:
        config_dir   = Path(__file__).parent.parent / 'config'
        config_files = sorted(config_dir.glob('config_*.yaml'), reverse=True)
        if config_files:
            config_path = config_files[0]

@timer_utils.timer 是一个装饰器（decorator），自动记录 main() 函数的运行时间。 **kwargs 让函数接受任意关键字参数，这里只用到了 config_path。

If no config path is provided, the program automatically finds the most recent YAML file in the config directory (sorted by filename in reverse order — timestamps in filenames make the latest file sort last alphabetically).

Solving 40+ ILP scenarios can take minutes. The cache skips recomputation if results already exist for the exact same config. Incremental caching means even an interrupted run can resume from where it left off.

类比: 就像考试前做过的历年真题答案——如果题目没变，直接查答案，不用再算一遍。

NoneScenario Loop and Incremental Saving (

start_idx = len(results)
for idx, scenario in enumerate(scenarios[start_idx:], start=start_idx):
    M = len(scenario['SVCs'])
    N = len(scenario['ECUs'])
    e_list = [ecu['capacity']    for ecu in scenario['ECUs']]
    n_list = [svc['requirement'] for svc in scenario['SVCs']]
    conflict_sets = scenario.get('conflict_sets', [])

    result = solve_service_deployment(M, N, e_list, n_list, conflict_sets)
    results.append(result)

    # Atomic write: write to .tmp first, then rename
    tmp = shared_cache_path.with_suffix(".tmp")
    with open(tmp, "w") as f:
        json.dump({"key": cache_key, "results": results}, f)
    tmp.replace(shared_cache_path)

Atomic write pattern: write to a temp file first, then atomically rename it. This ensures the cache file is never in a corrupted half-written state, even if the program crashes mid-write.

类比: 就像写支票时先在草稿纸上写好，确认没错再誊到正式支票上。
