1 什么是 DQN？ What is DQN?

核心思想 Core Concept

DQN（Deep Q-Network）是一种强化学习算法。智能体（Agent）在每一步选择一个动作（action），环境给它一个奖励（reward），智能体的目标是最大化长期总奖励。

DQN is a reinforcement learning algorithm. At each step the agent picks an action, the environment gives a reward, and the agent's goal is to maximise total long-term reward.

在这个项目里：

  • 动作 Action：选择把当前服务放到哪个 ECU（编号 0 到 N-1）
  • 状态 State / Observation：描述当前 ECU 剩余容量、当前服务需求等信息的向量
  • 奖励 Reward：合法放置得正分，违规得负分
  • 与 PPO/P4 的最大区别：DQN 没有动作屏蔽（Action Masking），违规动作仍然被执行，靠惩罚分数让 Q 网络自己学会避开非法动作

参数值含义（中）Meaning (EN)
DQN_LR 1e-3 学习率：神经网络每次更新的步伐大小。太大会震荡，太小会很慢。 Learning rate: step size for each network update. Too high = unstable; too low = slow.
DQN_BUFFER_SIZE 100,000 经验回放缓冲区容量：最多存储 10 万条 (状态, 动作, 奖励, 下一状态) 记录。 Replay buffer capacity: stores up to 100 K transitions (s, a, r, s') for random sampling.
DQN_LEARNING_STARTS 2,000 热身步数：前 2000 步完全随机探索，填充缓冲区，之后才开始训练。 Warm-up: first 2 000 steps are random, filling the buffer before training begins.
DQN_BATCH_SIZE 64 每次从缓冲区随机抽取 64 条经验来更新网络。 Mini-batch size: 64 transitions sampled randomly from the buffer per update.
DQN_TAU 1.0 目标网络更新方式：τ=1.0 是"硬复制"（完全覆盖），τ<1 是"软更新"（慢慢混合）。 τ=1.0 = hard copy (full overwrite of target network); τ<1 would be soft/polyak update.
DQN_GAMMA 0.99 折扣因子：未来奖励的权重。0.99 表示很重视未来，0 表示只看眼前。 Discount factor: weight of future rewards. 0.99 = strongly future-oriented.
DQN_TRAIN_FREQ 4 每走 4 步环境才更新一次网络，降低计算量。 Network update frequency: update once every 4 environment steps.
DQN_TARGET_UPDATE 500 每 500 步把在线网络的权重复制到目标网络，让训练目标更稳定。 Sync target network every 500 steps to stabilise training targets.
DQN_EXPLORATION_FRACTION 0.5 前 50% 的训练时间用于 ε-贪心探索（ε 从 1 逐渐降到 0）。 First 50% of training decays ε from 1.0 → 0.0 (exploration → exploitation).
DQN_EXPLORATION_FINAL_EPS 0.0 训练结束时 ε=0，完全贪心（总选 Q 值最高的动作）。 Final epsilon = 0: fully greedy at end of training.
DQN_NET_ARCH [128, 128] 神经网络结构：两层隐藏层，各 128 个神经元的 MLP（多层感知机）。 Network architecture: 2-hidden-layer MLP, each with 128 units.

3 env.py — 文档说明 Docstring & Design Philosophy

"""
DQN Environment — heavy penalty for violations + terminal bonus.

DQN (SB3) does not support action masking natively.  Constraint enforcement
is achieved via strong reward penalties.  The valid-action flag in the
observation lets the Q-network learn to avoid invalid actions.

Constraints:
    - Capacity violation → heavy penalty (-2.0), placement still proceeds
      (remaining_vms may go negative; visible in observation).
    - Conflict violation → heavy penalty (-2.0), placement still proceeds.
    - A well-trained agent should achieve zero violations on feasible scenarios.

Terminal bonus:
    +AR if zero violations, -AR if any violations.
"""
关键设计决定 Key Design Decision

无动作屏蔽（No Action Masking）：DQN 原版不支持动作屏蔽，所以即使选了违规的 ECU，放置仍然发生。环境靠 -2.0 的"痛感"让 Q 网络学会避开坏动作——就像学走路时摔跤了才知道不能那样走。

DQN does not support action masking. Violations still execute. The -2.0 penalty is the agent's "pain signal" — it must discover which actions are bad through trial and error.

4 DQNEnv.__init__() — 初始化

def __init__(self, ecus: list[ECU], services: list[SVC], scenarios=None):
    super().__init__()
    self._scenarios = scenarios   # 可选：训练场景列表
    self.ecus     = ecus           # N 个 ECU 对象
    self.services = services       # M 个 SVC 对象
    self.N = len(ecus)
    self.M = len(services)

    # 动作空间：从 0 到 N-1 选一个 ECU
    self.action_space = gym.spaces.Discrete(self.N)

    # 观测空间：形状 = 5N + 6 + M 的浮点向量，范围 [-1, 1]
    self.observation_space = gym.spaces.Box(
        low=-1.0, high=1.0, shape=(5 * self.N + 6 + self.M,), dtype=np.float32,
    )
动作空间 vs 观测空间 Action Space vs Observation Space
  • Discrete(N)：智能体每步只能选 0~N-1 中的一个整数（选哪个 ECU）
  • Box(shape=5N+6+M)：观测向量有 5N+6+M 个浮点数，描述当前状态
  • 例：N=10, M=15 → 观测维度 = 5×10+6+15 = 71

5 action_masks() — 没有屏蔽！

注意！Important!

DQNEnv 没有实现 action_masks() 方法（或如果实现则返回全 True）。这与 PPO/P4 的版本不同，P4 会直接禁止 Q 网络选择无效动作。

DQNEnv does NOT implement action masking. Every action is always "allowed" from the algorithm's perspective. The Q-network must learn to avoid bad actions purely through reward signals.

观测向量中包含 valid_flag（每个 ECU 是否有足够容量），作为"提示"。但这只是信息，不是强制约束：

# valid_flag in obs: 1.0 = enough capacity, 0.0 = not enough
valid_flag = (self.remaining_vms >= svc.requirement).astype(np.float32)
# 注意：仅检查容量（capacity），不检查冲突（conflict）！
设计含义 Design Implication

valid_flag 的 1/0 仅代表容量是否足够，不包含冲突约束。这意味着即使 valid_flag=1，放置到该 ECU 仍可能引发冲突惩罚。Q 网络需要同时学习容量和冲突两类约束。

6 _obs() — 观测向量构造 Observation Vector

def _obs(self) -> np.ndarray:
    max_cap   = float(np.max(self.initial_vms) + 1e-8)   # 防止除以零
    total_cap = float(np.sum(self.initial_vms) + 1e-8)

    # 当前待放服务的需求（归一化）
    svc = self.services[self._step]
    service_demand_norm = np.float32(svc.requirement / max_cap)

    # 冲突标志：对每个 ECU，1=放当前服务到这里会违反冲突约束
    conflict_flag = np.array([float(self._has_conflict(j, self._step))
                               for j in range(self.N)], dtype=np.float32)

    # 容量合法标志：1=该 ECU 有足够剩余容量
    valid_flag = (self.remaining_vms >= svc.requirement).astype(np.float32)

    return np.concatenate([
        [service_demand_norm],          # [0]       当前服务需求
        [self.ar],                      # [1]       当前 AR
        [remaining_usable_capacity_sum],# [2]       剩余可用总容量
        [remaining_service_demand_sum], # [3]       剩余服务总需求
        [remaining_usable_ecu_count],   # [4]       有足够容量的 ECU 比例
        [remaining_services_count],     # [5]       剩余服务数量比例
        initial_cap_pct,                # [6:6+N]   各 ECU 初始容量比例
        remaining_abs_norm,             # [6+N:6+2N]  剩余容量（可能为负！）
        conflict_flag,                  # [6+2N:6+3N] 冲突标志
        ecu_allowed_frac,               # [6+3N:6+4N] ECU 可接受服务比例
        valid_flag,                     # [6+4N:6+5N] 容量合法标志
        remaining_svcs,                 # [6+5N:6+5N+M] 剩余服务需求列表
    ]).astype(np.float32)

重要细节 Important Detail

remaining_abs_norm（索引 [6+N:6+2N]）在容量违规后可能为负数！这正是 DQN 无动作屏蔽的后果——ECU "超载"的信号仍然被保留在观测中，让网络看到"你之前犯了超载"。

7 step() — 奖励计算 Reward Calculation

def step(self, action: int):
    svc = self.services[self._step]   # 当前要放的服务

    # ① 检测违规
    cap_violated      = bool(self.remaining_vms[action] < svc.requirement)
    conflict_violated = self._has_conflict(action, self._step)

    # ② 计算惩罚（可叠加！同时违规 = -4.0）
    cap_penalty      = -2.0 if cap_violated      else 0.0
    conflict_penalty = -2.0 if conflict_violated else 0.0

    # ③ 利用率奖励：违规时 ru=0（不给正奖励）
    ru = 0.0 if (cap_violated or conflict_violated) else \
         svc.requirement / (self.initial_vms[action] + 1e-8)

    # ④ 无论违规与否，都执行放置（remaining_vms 可能变负！）
    self.remaining_vms[action] -= svc.requirement
    self.ecu_placements[action].add(self._step)
    self._update_ecu_allowed(action, self._step)

    # ⑤ 终局奖励（最后一步才触发）
    done = self._step >= self.M
    terminal_bonus = 0.0
    if done:
        terminal_bonus = self.ar if total_viol == 0 else -self.ar

    return self._obs(), float(ru + cap_penalty + conflict_penalty + terminal_bonus), ...

合法放置 Valid Placement

  • cap_penalty = 0
  • conflict_penalty = 0
  • ru = req / cap_initial（正数）
  • remaining_vms 正常减少
  • Total step reward = +ru

双重违规 Both Violated

  • cap_penalty = -2.0
  • conflict_penalty = -2.0
  • ru = 0.0
  • remaining_vms 变负！
  • Total step reward = -4.0
与 P4/PPO 的关键区别 Key Difference from P4

P4（PPO 版）使用动作屏蔽，违规动作根本不会被选到。DQN 版本即使违规也强制执行，remaining_vms 可以变成负数。这在观测向量中是可见的，让 Q 网络能"感受"到超载的后果。

8 终局奖励 Terminal Bonus

if done:
    # 零违规 → 额外奖励 +AR（鼓励整集无违规）
    # 有违规 → 额外惩罚 -AR（惩罚整集失败）
    terminal_bonus = self.ar if total_viol == 0 else -self.ar
为什么需要终局奖励？ Why Terminal Bonus?

DQN 的 Q 值是未来折扣奖励之和的估计。如果只有每步的小奖励，网络难以理解"整集都正确才算真正成功"。终局奖励 +AR 相当于给无违规的整集打一个"全集满分奖"，-AR 是"全集扣分"。

Terminal bonus acts as an episode-level signal. +AR rewards zero-violation episodes heavily; -AR punishes any violation. This is harsher than P4 (which gave +0.1×AR even with violations).

情况终局奖励含义
整集零违规 Zero violations+AR奖励：整集成功，获得等于当前 AR 的额外分
有任何违规 Any violation-AR惩罚：整集失败，扣除等于当前 AR 的分数

9 冲突约束处理 Conflict Constraint Logic

def _init_conflict_sets(self, K: int = 10) -> list[set]:
    # 随机生成 K 个冲突集，每个集包含 2~M 个服务
    # 同一冲突集内的服务不能放在同一 ECU 上
    sets = []
    for _ in range(K):
        j = random.randint(2, self.M)
        sets.append(set(random.sample(range(self.M), j)))
    return sets

def _has_conflict(self, ecu_idx: int, svc_idx: int) -> bool:
    # 返回 True 表示：把 svc_idx 放到 ecu_idx 会违反冲突约束
    return svc_idx not in self.ecu_allowed[ecu_idx]

def _update_ecu_allowed(self, ecu_idx: int, svc_idx: int) -> None:
    # 放置 svc_idx 后，从 ecu_allowed 中移除与它冲突的其他服务
    for subset in self.conflict_sets:
        if svc_idx in subset:
            self.ecu_allowed[ecu_idx] -= (subset - {svc_idx})
举例 Example

冲突集 = {SVC0, SVC2, SVC5}。如果把 SVC0 放到 ECU3，则 ECU3 的 ecu_allowed 里会删除 SVC2 和 SVC5。之后试图把 SVC2 放到 ECU3 时，_has_conflict 返回 True，触发 -2.0 惩罚。

10 为什么 LT 场景 DQN 还有违规？ Why DQN Still Violates in LT?

LT 场景（Large-scale Tight）：15 个服务，10 个 ECU，约束很紧。

-2.0 的惩罚在很多情况下不足以阻止 Q 网络犯错，原因如下：

  1. 冲突约束复杂：冲突关系随着放置历史动态变化，Q 网络很难从 0 开始学会所有冲突模式
  2. 无动作屏蔽：Q 网络没有外力阻止它选非法动作，必须完全靠负奖励自我约束
  3. 探索阶段产生大量违规：ε-贪心在前 50% 训练随机探索，产生大量违规样本存入缓冲区
  4. 稀疏信号：终局奖励只在最后一步给出，前期奖励信号弱，网络难以将某步违规与整集失败关联

11 run_all.py — DQN 模型构建 Model Construction

经验回放 Experience Replay

什么是经验回放？ What is Experience Replay?

每一步，智能体把 (状态, 动作, 奖励, 下一状态) 这个四元组存入一个"回放缓冲区"（Replay Buffer）。训练时随机从缓冲区抽取 64 条记录来更新网络，而不是用最新的连续几步。

Why? Consecutive steps are highly correlated. Random sampling breaks this correlation, making training more stable — like shuffling your homework problems before reviewing.

from stable_baselines3 import DQN

model = DQN(
    policy="MlpPolicy",     # 使用全连接神经网络（MLP）作为 Q 网络
    env=env,
    learning_rate=C.DQN_LR,           # 1e-3
    buffer_size=C.DQN_BUFFER_SIZE,     # 100,000 条经验
    learning_starts=C.DQN_LEARNING_STARTS, # 先随机探索 2000 步
    batch_size=C.DQN_BATCH_SIZE,       # 每次抽 64 条
    tau=C.DQN_TAU,                     # 1.0 = 硬复制目标网络
    gamma=C.DQN_GAMMA,                 # 0.99 折扣
    train_freq=C.DQN_TRAIN_FREQ,       # 每 4 步更新一次
    target_update_interval=C.DQN_TARGET_UPDATE, # 每 500 步同步目标网络
    exploration_fraction=C.DQN_EXPLORATION_FRACTION, # 前 50% 探索
    exploration_final_eps=C.DQN_EXPLORATION_FINAL_EPS, # 最终 ε=0
    policy_kwargs=dict(net_arch=C.DQN_NET_ARCH), # [128, 128]
    verbose=1,
)

ε-贪心探索 ε-Greedy Exploration

DQN 使用 ε-贪心（epsilon-greedy） 策略来平衡探索（Exploration）与利用（Exploitation）：

探索 Explore (ε→1.0～0.0)　前 50% 步数
利用 Exploit (ε=0.0)　后 50%
阶段ε 值行为
训练开始 Startε = 1.0完全随机选动作，大量探索环境
前 50% 训练ε 线性下降 1.0 → 0.0逐渐减少随机，增加依靠 Q 值决策
后 50% 训练ε = 0.0完全贪心：总选 Q 值最高的动作
# 每步决策：
if random() < epsilon:
    action = env.action_space.sample()   # 随机探索
else:
    action = argmax(Q(state))             # 贪心利用

双网络结构 Dual Network Structure

在线网络 Online Network

  • 每 4 步更新一次
  • 用于选择动作
  • 权重快速变化

目标网络 Target Network

  • 每 500 步从在线网络复制一次
  • 用于计算 TD 目标值
  • 权重缓慢更新，提供稳定训练目标
# DQN 的 Q 值更新公式（Bellman equation）：
# target = r + γ × max_a Q_target(s', a)
# loss = (Q_online(s, a) - target)²

# 注意：目标网络的 max_a 同时做了"选择"和"评估"两件事

12 reset() — 重置环境 Environment Reset

def reset(self, seed=None, options=None):
    super().reset(seed=seed)
    if self._scenarios is not None:
        # 随机抽取一个训练场景
        caps, reqs, _cs = random.choice(self._scenarios)
        self.ecus     = [ECU(f"ECU{i}", cap) for i, cap in enumerate(caps)]
        self.services = [SVC(f"SVC{i}", req) for i, req in enumerate(reqs)]

    # 服务按需求从大到小排序（先放大服务，类似 First Fit Decreasing）
    self.services = sorted(self.services, key=lambda s: s.requirement, reverse=True)

    self.remaining_vms   = self.initial_vms.copy()   # 重置剩余容量
    self.ecu_placements  = [set() for _ in range(self.N)]  # 清空放置记录
    self.ecu_allowed     = [set(range(self.M)) for _ in range(self.N)]  # 所有服务初始都允许
    self.ar              = 0.0
    self._step           = 0
    self.capacity_violations  = 0
    self.conflict_violations  = 0
    return self._obs(), {}
排序的意义 Why Sort Services?

将服务按需求从大到小排序后，智能体先处理"难放"的大服务，确保大服务有机会找到足够容量的 ECU。这类似于"First Fit Decreasing"装箱启发式算法，减少大服务最后无处可放的情况。
