# 实验方法说明 — ecu_lt_svc_p（ECU < SVC，N=10, M=15）

## 1. 问题定义

**任务**：将 M=15 个服务（SVC）依次分配到 N=10 个边缘计算单元（ECU），最大化平均资源利用率（AR），同时满足容量约束和冲突约束。

**目标函数（AR）**：

```
AR = (1 / |active ECUs|) × Σ_{i,j: x_{i,j}=1} req_i / cap_j
```

**约束**：
1. **容量约束**：∀j, Σ_{i: placed on j} req_i ≤ cap_j
2. **冲突约束**：∀ conflict_set C, ∀j, |{i ∈ C : placed on j}| ≤ 1

**lt 场景特点**：N=10 < M=15，服务数超过 ECU 数。每个 ECU 平均需承载 1.5 个服务，容量压力最大。ILP 最优 AR ≈ 0.74，是三个场景中最高的，因为密集打包导致每个 ECU 的利用率更高。场景生成时保证至少有一条大小 > N_ECUS 的冲突集，使部分约束理论上不可满足，测试方法在极端约束下的行为。

**lt 是最难场景**：
- 容量紧缺（15 个服务争抢 10 个 ECU 的空间）
- 更多冲突约束（冲突集大小可达 15）
- ILP 需要精细打包，RL 需要长程规划（15 步 episode）
- 无约束方法（P3、DQN/DDQN）在此场景违反率极高

---

## 2. 建模方式：顺序决策（MDP）

将放置问题建模为一个有限步 MDP：
- **状态** s_t：当前待放置服务的特征 + 所有 ECU 的剩余容量 + 冲突标记 + 已放置历史
- **动作** a_t：选择目标 ECU j ∈ {0, …, N-1=9}
- **奖励** r_t：放置当前服务后的 AR 增量
- **终止条件**：放置完所有 M=15 个服务（episode 固定长度 15 步，比 eq/gt 长 50%）

更长的 episode（15 步）要求 RL 具备更强的长程信用分配能力，这也是 GAMMA=0.999 的原因。

---

## 3. 六种方法详解

### 3.1 ILP 最优基线（problem2_ilp）

**性质**：整数线性规划，全局最优，非 RL。

**lt 场景特点**：决策变量数量增加（15×10=150 个 x_{i,j}），ILP 求解时间比 eq 略长，但仍在可接受范围内（每个 scenario < 1s）。

**lt 场景中**：ILP 最优 AR = 0.7361 ± 0.0128，零违反，全部 15 个服务均被成功放置。这是理论上界，RL 方法难以达到。

---

### 3.2 P3：无约束 PPO（problem3_ppo）

**约束处理**：完全无约束。

**lt 场景中严重失效**：
- AR = 11.84（远超理论最大值 1.0）：因为容量超载导致 req_i / cap_j 的分子异常大，AR 计算失真
- 476.6 次容量违反 + 428.6 次冲突违反（共 905 次违反 / 40 个 episode）
- 每个 episode 平均 60 次违反，完全不可用

P3 在 lt 场景揭示了无约束 RL 的根本缺陷：当资源紧缺时，策略学会了"疯狂超载"以最大化即时奖励，完全无视约束。

---

### 3.3 P4：动作掩码 PPO（problem4_ppo_mask）

**约束处理**：硬约束，动作掩码。

**掩码规则**：
1. ECU j 剩余容量 ≥ 当前服务需求
2. ECU j 未被当前冲突集中的其他服务占用

**lt 场景特点**：N=10 < M=15，后期（第 10~15 步）可能所有 ECU 都无法满足掩码条件（容量耗尽或全部冲突），此时无合法动作。环境通过跳过无法放置的服务来处理此情况（placed_mean = M = 15.0，说明 P4 仍能放置全部服务）。

**lt 场景中**：AR = 0.673，零违反，全部 15 个服务放置成功。AR 为 ILP 的 91.5%，gap 比 eq（98.8%）大，说明容量紧缺时掩码 PPO 更难找到最优打包方案。

---

### 3.4 P5：Lagrangian 约束松弛 PPO（problem5_ppo_lagrangian）

**约束处理**：软约束 + 自适应 λ 惩罚。

**奖励函数**：
```
r_t = (req_i / cap_j) - λ · c_t
```

**lt 场景特点**：资源紧缺使约束违反更频繁，λ 需要增大到更高值（接近 LAMBDA_MAX=5.0）才能控制违反率。但高 λ 会压低 AR，形成更强的 AR-约束 tradeoff。

**lt 场景中**：AR = 0.701（最高的可行 RL 方法，为 ILP 的 95.2%），但仍有 5.0 次容量违反和 37.6 次冲突违反（违反率 8.9%）。这表明在极端约束下，Lagrangian 方法的对偶上升收敛较慢，5M 步训练不足以完全消除违反。

---

### 3.5 P6：PPO + 局部修复启发式（problem6_ppo_opt）

**约束处理**：后处理修复，冲突发生时 Best-Fit 重新定位；若无合法备选，跳过该服务。

**lt 场景特点**：N=10 < M=15，修复空间受限。当 RL 策略频繁选择被占用的 ECU 时，修复可能找不到合法备选，导致服务被放弃（placed_mean = 12.89 < 15）。

**lt 场景中**：AR = 0.679，12.89 个服务（平均丢失 2.11 个），4.6 次容量违反，1.0 次冲突违反。P6 是唯一 placed_mean < M 的方法，说明在 lt 场景中修复启发式会主动放弃部分服务以满足冲突约束。

---

### 3.6 DQN / DDQN（problem_dqn / problem_ddqn）

**约束处理**：违反终止 + 负奖励（r = -1）。

**lt 场景严重失效**：
- DQN：AR=0.667，68.1% 违反率，21.8 次容量违反，31.8 次冲突违反
- DDQN：AR=0.659，64.4% 违反率，20.6 次容量违反，27.8 次冲突违反

在 M=15 步的长 episode 中，终止惩罚策略使得策略无法完成所有服务的放置——episode 频繁在第 3~5 步就被终止，导致大量不完整的放置尝试。经验回放中充满了被截断的轨迹，Q 值估计不准，学习效率极低。

**DDQN 略优于 DQN**：双网络结构降低了 Q 值高估，减少了贪心选择到违反动作的频率，但本质问题（终止惩罚在紧缺场景无效）未解决。

## 4. 实验执行流程

### 4.1 单次单 seed 执行

```
TRAIN_SEED=0 python ecu_lt_svc_p/{problem}/run_all.py
```

每个 `run_all.py` 执行：
1. 加载 YAML 配置（N=10 ECU，M=15 SVC 的 200 个 scenario）
2. 按 SEED 划分 160 训练 / 40 测试 scenario
3. 对测试 scenario 逐一运行 ILP（15 变量，约束更复杂）
4. 训练 RL（N_ENVS=40，episode 长度 M=15）
5. 对 40 个测试 scenario 各评估 1 个 episode
6. 生成 `results/summary.csv`、`results/results.json`、图表

### 4.2 多 seed 并行执行

```bash
bash scripts/run_multi_seed.sh --seeds "0 1 2 3 4" --total-timesteps 5000000
```

lt 环境与 eq、gt 同时并行。由于 M=15 episode 更长，lt 的训练通常比 eq/gt 慢约 50%。

### 4.3 结果汇总

`aggregate_results.py` 对 5 个 seed 的 lt 组结果计算均值±标准差。

## 5. lt 场景方法比较要点

| 方法 | AR | %ILP | 违反率 | 容量违反 | 冲突违反 | 服务放置数 |
|------|-----|------|--------|----------|----------|------------|
| ILP | 0.736 | 100% | 0.0% | 0 | 0 | 15.0 |
| P3 | 11.84 | — | 188.6% | 477 | 429 | 15.0 |
| P4 | 0.673 | 91.5% | 0.0% | 0 | 0 | 15.0 |
| **P5** | **0.701** | **95.2%** | 8.9% | 5 | 38 | 15.0 |
| P6 | 0.679 | 92.3% | 1.2% | 5 | 1 | 12.9 |
| DQN | 0.667 | 90.6% | 68.1% | 22 | 32 | 15.0 |
| DDQN | 0.659 | 89.5% | 64.4% | 21 | 28 | 15.0 |

**lt 场景核心结论**：
1. P3、DQN、DDQN 在资源紧缺场景彻底失效，违反率极高
2. P4（掩码）是最稳健的方法，保证零违反但 AR gap 最大
3. P5（Lagrangian）在 lt 场景取得最高的可行 AR（0.701），但仍有一定违反
4. P6 以牺牲放置完整性（丢失 2.1 个服务）换取低违反率
5. lt 场景是区分不同约束策略能力的最佳压力测试环境
