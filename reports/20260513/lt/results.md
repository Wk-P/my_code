# 结果数据说明 — ecu_lt_svc_p（ECU < SVC，N=10, M=15）

## 1. 数据来源

`results.csv` 中的数据来自 `logs/multi_seed_results/aggregate_summary.csv`，为 5 个独立随机种子（seed=0,1,2,3,4）运行结果的均值和标准差。

每个 seed 对应一次完整的训练+评估流程：
- 训练集：160 个 scenario（80%）
- 测试集：40 个 scenario（20%），用于计算最终指标

---

## 2. 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `group` | str | 实验环境组名，本文件均为 `ecu_lt_svc_p` |
| `problem` | str | 问题编号 |
| `method` | str | 方法显示名称 |
| `n_seeds` | int | 参与统计的种子数量（均为 5） |
| `ar_mean_avg` | float | 跨 5 个 seed 的平均 AR（平均资源利用率） |
| `ar_mean_std` | float | AR 的跨 seed 标准差 |
| `placed_mean_avg` | float | 平均成功放置的服务数（最大值 M=15） |
| `placed_mean_std` | float | 放置数的跨 seed 标准差 |
| `viol_rate_avg` | float | 平均违反率（每步产生违反的比例） |
| `viol_rate_std` | float | 违反率标准差 |
| `cap_viol_avg` | float | 平均容量违反次数（40 个测试 episode 总计） |
| `cap_viol_std` | float | 容量违反标准差 |
| `conflict_viol_avg` | float | 平均冲突违反次数（40 个测试 episode 总计） |
| `conflict_viol_std` | float | 冲突违反标准差 |

**特别注意**：
- P3 的 `ar_mean_avg` = 11.84，这是计算异常值（允许容量超载后 AR 公式失真），不代表真实利用率
- P6 的 `placed_mean_avg` = 12.89 < M=15，说明修复启发式放弃了部分无法放置的服务
- P6 的 `viol_rate_avg` 为空（NaN），因为修复逻辑绕过了违反率统计

## 3. 各方法数据解读

### ILP（最优基线）
- **AR = 0.7361 ± 0.0128**：lt 场景最高 ILP AR，因为密集打包（15服务/10ECU）使每个 ECU 利用率更高
- **零违反，全部 15 个服务成功放置**
- 标准差（0.013）介于 eq（0.005）和 gt（0.017）之间

### P3 — 无约束 PPO（严重失效）
- **AR = 11.84 ± 0.404**：计算失真，无意义
- **容量违反 = 476.6 次**（40 个 episode，平均每集 11.9 次容量违反）
- **冲突违反 = 428.6 次**（平均每集 10.7 次冲突违反）
- **违反率 = 188.6%**（>1 是因为每步平均有 1.9 个违反）
- lt 场景中 P3 的违反远超 eq（容量：477 vs 0）和 gt（容量：0 vs 0，冲突：429 vs 112）
- **结论**：在资源紧缺场景，无约束 RL 彻底失效，不仅无用，还会产生灾难性的资源超载

### P4 — 动作掩码 PPO
- **AR = 0.6727 ± 0.0118**，为 ILP 的 **91.4%**
- **零违反，全部 15 个服务成功放置**
- AR gap（8.6%）是三个场景中最大的（eq: 1.3%，gt: 2.3%）
- **原因**：N=10 < M=15，后期 ECU 几乎被填满，可行动作集大幅缩小，策略难以选择最优 ECU
- **结论**：最安全的方法（零违反），但在 lt 场景与 ILP 的 AR 差距最明显

### P5 — Lagrangian PPO
- **AR = 0.7006 ± 0.0143**，为 ILP 的 **95.2%**，**三个可行 RL 方法中 AR 最高**
- 违反率 = 8.9%（高于 eq 的 0.15%）
- 容量违反 = 5.0 次，冲突违反 = 37.6 次
- **原因**：lt 场景冲突更频繁，λ 需要上升到较高值，但 5M 步训练不足以完全消除违反
- **结论**：在 lt 场景 AR 最高但存在明显违反，适合对 AR 要求高、可容忍少量违反的场景

### P6 — PPO + 修复启发式
- **AR = 0.6792 ± 0.0098**，为 ILP 的 **92.3%**
- **placed_mean = 12.89**（平均丢失 2.11 个服务，约 14.1% 服务无法放置）
- 容量违反 = 4.6 次，冲突违反 = 1.0 次
- 标准差最小（0.0098），跨 seed 最稳定
- **原因**：当修复失败时，宁可放弃服务也不产生冲突违反；容量违反来自未触发修复逻辑的超载
- **结论**：牺牲放置完整性换取低冲突违反；在要求严格冲突约束满足但允许服务丢弃的场景有用

### DQN（严重失效）
- **AR = 0.6670 ± 0.0137**，为 ILP 的 **90.6%**
- **违反率 = 68.1%**（每 3 步约有 2 步产生违反）
- 容量违反 = 21.8 次，冲突违反 = 31.8 次
- 标准差（0.014）偏大
- **原因**：15 步 episode 中终止惩罚导致大量截断，经验回放中充斥无效轨迹

### DDQN（严重失效）
- **AR = 0.6591 ± 0.0184**，为 ILP 的 **89.5%**
- **违反率 = 64.4%**
- 容量违反 = 20.6 次，冲突违反 = 27.8 次
- 标准差最大（0.0184），最不稳定
- DDQN 略优于 DQN（违反率低 3.7%），但本质问题相同

## 5. 与 eq/gt 场景的跨场景对比

| 方法 | eq AR (% ILP) | gt AR (% ILP) | lt AR (% ILP) | 趋势 |
|------|---------------|---------------|---------------|------|
| ILP | 0.5418 (100%) | 0.6301 (100%) | 0.7361 (100%) | AR随紧缺程度升高 |
| P4 Mask | 98.7% | 97.7% | 91.4% | 紧缺时掩码效果明显下降 |
| P5 Lagrange | 99.1% | 97.9% | 95.2% | 紧缺时仍保持最高RL AR |
| P6 Repair | 99.1% | 98.0% | 92.3% | 紧缺时修复失败，服务丢弃 |
| DQN | 97.4% | 96.8% | 90.6% | 紧缺时严重退化 |
| DDQN | 98.1% | 96.8% | 89.5% | 紧缺时最差 |
| P3 冲突违反 | 143.4 | 111.6 | 428.6 | lt场景爆炸式增长 |
| DQN 违反率 | 5.5% | 6.5% | 68.1% | lt场景急剧恶化 |

**核心规律**：随资源紧缺程度从 gt → eq → lt：
1. ILP 最优 AR 升高（密集打包带来更高利用率）
2. 所有 RL 方法的 AR / ILP 比例下降（紧缺场景更难优化）
3. 无约束方法的违反率急剧上升（lt 是分水岭）
4. P5（Lagrangian）在三个场景中始终保持最高 RL AR，是最鲁棒的 RL 方法

---

## 6. 数据可靠性说明

- **n_seeds = 5**：所有方法在 5 个独立种子下重复
- **lt 场景标准差**：通常介于 eq 和 gt 之间（0.010~0.018），方法间差异显著
- **P3 数据异常**：AR=11.84 是计算值（非实际利用率），违反率 > 1 也是正常计算结果（每步可能产生多次违反计数）
- **P6 placed_mean 标准差**：0.1091，说明不同 seed 下放弃的服务数略有不同（12.78~13.00）
