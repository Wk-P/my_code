# 实验方法说明 — ecu_gt_svc_p（ECU > SVC，N=15, M=10）

## 1. 问题定义

**任务**：将 M=10 个服务（SVC）依次分配到 N=15 个边缘计算单元（ECU），最大化平均资源利用率（AR），同时满足容量约束和冲突约束。

**目标函数（AR）**：

```
AR = (1 / |active ECUs|) × Σ_{i,j: x_{i,j}=1} req_i / cap_j
```

**约束**：
1. **容量约束**：∀j, Σ_{i: placed on j} req_i ≤ cap_j
2. **冲突约束**：∀ conflict_set C, ∀j, |{i ∈ C : placed on j}| ≤ 1

**gt 场景特点**：N=15 > M=10，资源供大于求。ECU 数比服务数多 5 个，平均每个 ECU 承载不足 1 个服务。ILP 最优 AR ≈ 0.63，高于 eq 场景（更多 ECU 提供更多放置选项，倾向于高容量利用的 ECU）。冲突约束相对容易满足（ECU 数量充足），但策略需要学会"聚集"放置（将服务集中到少数 ECU 以提升 AR）。

---

## 2. 建模方式：顺序决策（MDP）

将放置问题建模为一个有限步 MDP：
- **状态** s_t：当前待放置服务的特征 + 所有 ECU 的剩余容量 + 冲突标记 + 已放置历史
- **动作** a_t：选择目标 ECU j ∈ {0, …, N-1=14}（动作空间比 eq/lt 更大）
- **奖励** r_t：放置当前服务后的 AR 增量
- **终止条件**：放置完所有 M=10 个服务（episode 固定长度 10 步）

---

## 3. 六种方法详解

### 3.1 ILP 最优基线（problem2_ilp）

**性质**：整数线性规划，全局最优，非 RL。

**流程**：
1. 对每个测试 scenario，用 PuLP + CBC 求解器直接求解 ILP
2. 决策变量 x_{i,j} ∈ {0,1}，j ∈ {0,...,14}（15 个 ECU）
3. 得到全局最优放置方案和对应 AR

**gt 场景中**：ILP 最优 AR = 0.6301 ± 0.0172，零违反。标准差（0.017）比 eq（0.005）大，因为 15 个 ECU 的排列组合使 scenario 间差异更大。

---

### 3.2 P3：无约束 PPO（problem3_ppo）

**约束处理**：无约束，剩余容量可为负。

**gt 场景特点**：由于 ECU 数量充裕（N=15），容量约束极少被触发，主要违反来自冲突约束。AR 虚高（0.874）是因为允许无限超载。

**gt 场景中**：AR = 0.873，111.6 次冲突违反，0 次容量违反（资源充裕，容量从不超载）。

---

### 3.3 P4：动作掩码 PPO（problem4_ppo_mask）

**约束处理**：硬约束，动作掩码屏蔽所有非法 ECU。

**掩码规则**：
1. ECU j 剩余容量 ≥ 当前服务需求
2. ECU j 未被当前冲突集中的其他服务占用

**gt 场景特点**：N=15 提供更多可行动作，掩码后通常仍有多个合法选项，策略有较大的优化空间。偶发的服务未放置（placed_mean=9.98）可能因场景中存在极端冲突约束。

**gt 场景中**：AR = 0.616，零违反。AR 为 ILP 的 97.7%。

---

### 3.4 P5：Lagrangian 约束松弛 PPO（problem5_ppo_lagrangian）

**约束处理**：软约束 + 自适应 λ 惩罚。

**奖励函数**：
```
r_t = (req_i / cap_j) - λ · c_t
```

**λ 更新**（每 20 个 episode）：
```
λ ← clip(λ + 0.005 × (avg_viol_rate - 0.0), 0.0, 5.0)
```

**gt 场景中**：AR = 0.617，1.6 次冲突违反。λ 惩罚能有效收敛到低违反，同时保持接近最优的 AR。

---

### 3.5 P6：PPO + 局部修复启发式（problem6_ppo_opt）

**约束处理**：后处理修复，冲突发生时 Best-Fit 重新定位。

**gt 场景中**：AR = 0.618，7.0 次容量违反，1.0 次冲突违反。容量违反比 eq 更高（7.0 vs 2.8），可能是因为 N=15 时修复逻辑有更多备选 ECU，部分备选 ECU 容量较小，触发超载。

---

### 3.6 DQN / DDQN（problem_dqn / problem_ddqn）

**约束处理**：违反终止 + 负奖励。

**gt 场景特点**：N=15 的大动作空间（Discrete(15)）对 DQN 不利，Q 值估计维度更高，收敛更慢。

**gt 场景中**：DQN AR=0.610，4.5% 违反率；DDQN AR=0.610，3.5% 违反率。两者性能几乎相同，高于 eq 场景但违反率仍高于 P4/P5。

---

## 4. 实验执行流程

### 4.1 单次单 seed 执行

```
TRAIN_SEED=0 python ecu_gt_svc_p/{problem}/run_all.py
```

每个 `run_all.py` 执行：
1. 加载 YAML 配置 → 解析 200 个 scenario
2. 按 SEED 划分 160 训练 / 40 测试 scenario
3. 对测试 scenario 逐一运行 ILP 求解
4. 训练 RL 算法（N_ENVS=40 并行，到达 TOTAL_STEPS）
5. 对 40 个测试 scenario 各评估 1 个 episode
6. 生成 `results/summary.csv`、`results/results.json`、训练曲线和对比图

### 4.2 多 seed 并行执行

```bash
bash scripts/run_multi_seed.sh --seeds "0 1 2 3 4" --total-timesteps 5000000
```

gt 环境与 eq、lt 同时并行运行（3 个环境组同时启动）。结果归档到 `logs/multi_seed_results/seed_{seed}/ecu_gt_svc_p/`。

### 4.3 结果汇总

`aggregate_results.py` 对 5 个 seed 的 gt 组结果计算均值±标准差，输出至 `aggregate_summary.csv`。

## 5. gt 场景方法比较要点

| 方法 | 关键特性 | gt 场景优势 | gt 场景劣势 |
|------|----------|-------------|-------------|
| ILP | 全局最优 | 上界基准 | 计算复杂度高（NP-hard） |
| P3 | 无约束 | AR 最高 | 111 次冲突违反，不可用 |
| P4 | 动作掩码 | 零违反，97.7% ILP | 少量服务未放置 |
| P5 | Lagrangian | 近零违反，97.9% ILP | 偶发 1.6 次冲突违反 |
| P6 | 修复启发式 | AR 最高（98.0% ILP） | 7 次容量违反 |
| DQN | off-policy | 3M 步训练 | 4.5% 违反率 |
| DDQN | 双网络 | 略低高估偏差 | 3.5% 违反率 |

gt 场景中，P5（Lagrangian）和 P6（修复）的差距缩小，均接近 ILP 的 98%，说明资源充裕时多种约束策略都能有效工作。
